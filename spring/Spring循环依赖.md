这是一个被问烂了但依然能难倒不少人的经典面试题。它不仅考验你对 Spring Bean 生命周期的理解，还考验你对“中间层”思想的掌握。

---

### 1\. 核心结论 (Core Conclusion)

**循环依赖是指两个或多个 Bean 之间相互持有对方的引用（如 A 依赖 B，B 又依赖 A）。** 如果不处理，会导致 Bean 创建过程陷入死循环。Spring 通过\*\*“三级缓存”机制**和**“提前暴露半成品对象”\*\*的策略，完美解决了单例（Singleton）模式下的 Setter 注入循环依赖。

---

### 2\. 深度解析 (Detailed Breakdown)

#### ① 为什么会发生问题？（先有鸡还是先有蛋）

在 Spring 中，创建一个 Bean 流程大致是：**实例化 -> 属性填充 -> 初始化**。

1.  Spring 尝试创建 A，发现 A 依赖 B，于是去创建 B。
    
2.  创建 B 时，发现 B 依赖 A。
    
3.  如果此时 A 还没有创建好，B 就会一直等 A，A 也在等 B 结束，最终导致 `BeanCurrentlyInCreationException` 异常。
    

#### ② Spring 的救星：三级缓存

Spring 在 `DefaultSingletonBeanRegistry` 类中定义了三个 Map：

-   **一级缓存 (**`singletonObjects`**)：** 存放**完全初始化好**的成品 Bean（可以直接用的）。
    
-   **二级缓存 (**`earlySingletonObjects`**)：** 存放**半成品** Bean（实例化了但还没填充属性）。主要用于解决 AOP 代理对象的循环依赖。
    
-   **三级缓存 (**`singletonFactories`**)：** 存放 **Bean 工厂**。它不直接存对象，而是存一个“工厂”，用来在需要时生成代理对象或原始对象。
    

#### ③ 解决流程（以 A、B 互相依赖为例）

1.  **创建 A：** A 实例化后，先把自己的 `ObjectFactory`（三级缓存）存起来。
    
2.  **A 注入 B：** 发现需要 B，去容器找 B，没找到，开始创建 B。
    
3.  **创建 B：** B 实例化后，也把自己的工厂存入三级缓存。
    
4.  **B 注入 A：**
    

-   B 去找 A，发现一级、二级缓存都没有。
    
-   但在**三级缓存**里找到了 A 的工厂。
    
-   B 调用工厂获取 A 的引用（如果是 AOP 此时拿到的是代理对象，否则是原始对象），并将 A 放入**二级缓存**。
    

5.  **B 完成：** B 填充完 A 的引用，完成初始化，进入**一级缓存**。
    
6.  **A 完成：** A 拿到 B 的成品，继续完成初始化，进入**一级缓存**。
    

-   **形象类比：** 盖房子。A 还没装修（属性填充）就先把毛坯房（实例化）的钥匙挂在墙上（三级缓存）。B 需要 A 时，先拿毛坯房钥匙进去装修自己，等 B 盖好了，A 再拿着 B 的成品钥匙完成最后的装修。

### 3\. 面试官视角：追问与陷阱 (Interviewer's Lens)

#### 常见追问：

1.  **为什么需要三级缓存？二级缓存不行吗？**
    

-   *核心回答：* **为了处理 AOP（动态代理）**。如果 A 需要被代理，二级缓存只能存原始对象。通过三级缓存的 `ObjectFactory`，Spring 可以在 B 需要 A 的那一刻，动态地生成 A 的代理对象并交给 B，从而保证 B 依赖的是代理后的 A。
    

2.  **构造器注入（Constructor Injection）能解决循环依赖吗？**
    

-   *核心回答：* **默认不行**。因为构造器注入在“实例化”阶段就发生了，此时连三级缓存都还没来得及存。*解法：* 在构造参数上加 `@Lazy`。
    

#### 核心坑点 (Pitfalls)：

-   **非单例（Prototype）不支持：** Spring 不缓存原型 Bean，所以无法解决其循环依赖，直接报错。
    
-   **内存溢出风险：** 虽然 Spring 解决了循环依赖，但过度的循环引用说明架构设计不合理（类职责不清晰），应该通过引入中间层或观察者模式来重构。

### 4\. 代码演示 (Code Snippet)

```java
@Component
public class A {
    @Autowired
    private B b; // Setter 注入，Spring 可通过三级缓存解决
}

@Component
public class B {
    @Autowired
    private A a;
}

// ---------------------------
// 失败案例：构造器注入
@Component
public class C {
    private D d;
    public C(D d) { this.d = d; } // 这里会直接抛异常
}

@Component
public class D {
    private C c;
    public D(C c) { this.c = c; }
}
```

### 5\. 总结 (Summary)

-   **本质：** 利用“提前暴露半成品”来打破死锁。
    
-   **结构：** 依靠三级 Map 缓存协作。
    
-   **局限：** 只支持单例模式下的 Setter 注入，不支持构造器注入（除非 `@Lazy`）。
