这是一个非常经典、在真实项目中极易引发争议的“微观排错题”。很多候选人对“有没有事务”的理解往往停留在表面，没有结合 Spring AOP 和底层 JDBC `Connection` 去思考。

---

### 1\. 核心结论 (Core Conclusion)

如果一个**有事务的方法（A）内部调用了一个没有事务的方法（B）**，**方法 B 会“被动加入”方法 A 的事务中，和方法 A 共享同一个数据库连接（Connection），受方法 A 的统一提交或回滚控制。**

**一句话总结：** “大哥（A）罩着小弟（B）”。不管小弟 B 身上有没有 `@Transactional` 注解，只要它被大哥 A 叫来干活，它执行的 SQL 都会挂在大哥 A 的事务账本上，同生共死。

---

### 2\. 深度解析 (Detailed Breakdown)

要彻底理解这个问题，我们必须回到上一题讲过的 **Spring 事务底层原理（ThreadLocal 绑定 Connection）**。

#### 2.1 为什么 B 会被拉进 A 的事务？

1.  **方法 A 被调用：** Spring AOP 拦截器发现 A 有 `@Transactional`。
    
2.  **开启事务：** 拦截器找事务管理器拿了一个 `Connection`（假设叫 Conn-1），执行 `Conn-1.setAutoCommit(false)`，然后**把 Conn-1 绑定到当前线程的 ThreadLocal 里**。
    
3.  **方法 A 开始执行：** A 里面执行了一条 SQL，MyBatis/JdbcTemplate 乖乖去 ThreadLocal 里拿到 Conn-1 执行。
    
4.  **A 调用了方法 B：** 方法 B 身上没有注解，所以 Spring AOP 根本不会拦截 B，也不会为 B 做任何特殊处理。B 就是一个极其普通的 Java 方法。
    
5.  **B 内部执行 SQL：** 当 B 里的 MyBatis/JdbcTemplate 要执行 SQL 时，它们依然会去**当前线程的 ThreadLocal** 里找连接。
    

-   *结果：* 它们神奇地发现了大哥 A 刚才放进去的 **Conn-1**！
    
-   于是，B 的 SQL 也顺理成章地用 Conn-1 执行了。
    

6.  **最后收尾：** 方法 A 执行完毕，AOP 拦截器统一对 Conn-1 执行 `commit` 还是 `rollback`。因为 A 和 B 都在 Conn-1 上执行操作，所以它们同生共死。
    

#### 2.2 变种情况：B 抛出异常了怎么办？

-   **情况 1：A 没有** `catch` **B 的异常。** 异常一路抛给 Spring AOP 拦截器，拦截器一看“哦吼，出异常了”，直接把 Conn-1 **回滚 (rollback)**。A 和 B 的操作全部作废。
    
-   **情况 2：A 用** `try-catch` **把 B 的异常吃掉了。** A 正常执行完毕，拦截器准备提交。此时因为 B 只是个普通方法（没有 AOP 代理去干预），它不会悄悄把连接标记为 `rollback-only`（回顾上一题的陷阱）。因此，**A 和 B 之前成功执行的 SQL 会被一起提交 (commit)**。

### 3\. 面试官视角 (The Interviewer's Lens)

**高频追问与陷阱（考察你的边界意识）：**

1.  **陷阱：如果方法 B（没事务）是另一个 Service 的方法，A 调用 B 还会加入事务吗？**
    

-   *满分回答：* **依然会加入！** 因为 Spring 事务依赖的是\*\*线程（ThreadLocal）\*\*而不是类。只要 A 和 B 运行在同一个线程里，B 的持久层框架就能从 ThreadLocal 中拿到 A 绑定的那个 `Connection`。
    

2.  **追问：既然有没有** `@Transactional` **都会加入外层事务，那为什么还要在内层方法 B 上加** `@Transactional(propagation = Propagation.REQUIRED)`**？**
    

-   *满分回答：* 为了**语义明确**和**独立使用的安全**。
    
-   如果 B 永远只被 A 调用，加不加无所谓。
    
-   但如果有一天，别人单独调用了 B 呢？如果 B 没加注解，它就会在非事务状态下运行（自动提交，每条 SQL 独立），极易产生脏数据。加上 `REQUIRED` 就是告诉 Spring：“如果有现成的事务我加入，如果没有，你必须给我建一个！”
    

3.  **架构思维考察：把所有方法都加上事务有什么坏处？**
    

-   *满分回答：* 数据库连接是极其宝贵的资源。如果在只读方法（纯 `SELECT`）或者包含大量网络调用的方法上加事务，会导致数据库连接被长时间占用（长事务），甚至引发死锁或连接池耗尽。没必要加事务的地方，坚决不加。

### 4\. 代码演示 (Code Snippet)

这是一个最典型的“大哥罩小弟”的场景：

```java
@Service
public class OrderService {
    
    @Autowired
    private InventoryService inventoryService;

    // 大哥 A：带有事务
    @Transactional(rollbackFor = Exception.class)
    public void createOrder() {
        // 1. A 执行 SQL
        jdbcTemplate.update("INSERT INTO t_order (id) VALUES (1)");
        
        // 2. 调用小弟 B
        inventoryService.decreaseStock();
        
        // 3. 模拟 A 报错
        int x = 1 / 0; 
        
        // 结果：因为 A 报错回滚了底层的 Connection，
        // 小弟 B 刚才执行的 UPDATE 语句也会跟着一起回滚！
    }
}

@Service
public class InventoryService {
    
    // 小弟 B：没有任何事务注解
    public void decreaseStock() {
        // 当执行这句 SQL 时，JdbcTemplate 会从 ThreadLocal 里拿到大哥 A 的 Connection
        jdbcTemplate.update("UPDATE t_stock SET count = count - 1 WHERE id = 1");
    }
}
```

---

### 5\. 总结 (Summary)

-   **核心法则：** 只要在同一个线程内，底层 SQL 执行器就会去 ThreadLocal 找 Connection。
    
-   **被动加入：** 无事务方法 B 调用的 SQL，会被迫挂载到有事务方法 A 开启的连接上。
    
-   **生死与共：** B 会跟着 A 一起提交或一起回滚（前提是异常没有被吃掉）。
