
这是一个非常经典的**系统设计与网络架构**面试题。面试官通过这个问题，可以考察你从“简单逻辑”到“分布式一致性”再到“性能调优”的综合思考能力。

---

### 💡 核心结论
负载均衡（Load Balancing）算法的核心目标是：**将请求合理分配到后端服务器，以实现资源利用率最大化、响应时间最小化、并保证系统的高可用性。**

常用算法可以分为三大类：**静态算法**（不看服务器状态）、**动态算法**（看服务器压力）、**哈希算法**（看请求特征）。

---

### 🔍 深度原理解析 (Deep Dive)

我们将从**原理、实现思路、适用场景**三个维度展开：

#### 1. 轮询类 (Round Robin) —— 最简单的“排排坐，吃果果”
* **普通轮询 (Round Robin)**:
 * **实现**：维护一个简单的计数器 `i`，每次请求来了执行 `i = (i + 1) % n`。
 * **场景**：后端服务器硬件配置完全相同，且业务请求的处理耗时基本一致。
* **加权轮询 (Weighted Round Robin)**:
 * **实现**：给每台机器设置权重（Weight）。比如 A(3), B(1)，序列就是 A, A, A, B。
 * **场景**：**新老机器混用**。比如 16核服务器权重设为 10，4核服务器权重设为 2。

#### 2. 最小连接数类 (Least Connections) —— “谁闲着谁干”
* **实现**：负载均衡器记录每台服务器当前正在处理的活动连接数（Active Connections），新请求发给连接数最少的那台。
* **加权最小连接**：在连接数的基础上乘以权重系数。
* **场景**：**长连接业务**（如数据库连接、WebSocket、耗时较长的文件上传）。
 * *理由*：轮询可能导致某台机器虽然请求多，但全是短请求；而另一台机器请求少，但全是长请求导致过载。最小连接数能更公平地反映“忙碌程度”。

#### 3. 哈希类 (Hash) —— “从哪来，回哪去”
* **源地址哈希 (Source IP Hash)**:
 * **实现**：`hash(Client_IP) % n`。
 * **场景**：需要 **Session 保持**。比如某个用户登录后，必须访问同一台服务器才能拿到本地 Session（虽然现在多用 Redis 共享 Session，但 IP Hash 依然是简单有效的备选方案）。
* **一致性哈希 (Consistent Hashing) —— 进阶高分项 🌟**:
 * **实现**：构建一个 0 到 2^32-1 的哈希环。服务器和请求都映射到环上，顺时针寻找最近的节点。
 * **场景**：**分布式缓存（Redis/Memcached）**。
 * **优势**：当某台服务器宕机或扩容时，**只会影响一小部分数据**，而普通取模会导致全量缓存失效（缓存雪崩）。

### 🧐 面试官视角 (The "Interviewer's Lens")

面试官通常会通过以下追问来区分“背诵者”和“实践者”：

#### 常见追问与避坑 (Follow-ups & Pitfalls)

1. **高级追问：“如果某台机器性能突然变慢（比如磁盘 IO 爆了），但连接数还没满，轮询和最小连接数都失效了怎么办？”（高频 🌟）**
 * *满分回答*：引入 **最快响应时间 (Least Response Time)** 算法。
 * 负载均衡器不仅看连接数，还实时监测后端服务器的 **响应延迟 (RT)**。谁回得快，谁拿到的请求就多。这能自动绕过“亚健康”状态的服务器。

2. **工程实践：“Nginx 默认是什么算法？gRPC 呢？”**
 * **Nginx**：默认是轮询（Round Robin）。
 * **gRPC**：客户端负载均衡通常默认为 `pick_first`（选第一个），可以配置为轮询。

3. **避坑指南：权重设置的陷阱**
 * 权重不应只看 CPU。如果业务是内存密集型，即便 CPU 空闲，内存满了也会导致 OOM。优秀的系统会根据 CPU、内存、网络 IO 的多维度健康检查来动态调整权重。

### 💻 极客实践：用 Go 简单实现一个加权轮询

```go
type Server struct {
    Addr   string
    Weight int
}

func WeightedRoundRobin(servers []Server) string {
    totalWeight := 0
    for _, s := range servers {
        totalWeight += s.Weight
    }
    
    // 模拟计数器，实际应用中需加锁或用 Atomic
    count = (count + 1) % totalWeight 
    
    currentWeight := 0
    for _, s := range servers {
        currentWeight += s.Weight
        if count < currentWeight {
            return s.Addr
        }
    }
    return ""
}
```

### 📝 一分钟速记总结 (Summary)

1. **选轮询**：环境单纯，机器配置一样，追求高性能分配。
2. **选最小连接**：处理请求耗时差异大，或者有长连接。
3. **选源 IP 哈希**：需要简单的状态保持（Session）。
4. **选一致性哈希**：分布式缓存、扩容缩容频繁的场景。
5. **选最快响应**：对延迟极度敏感，追求极致用户体验。
