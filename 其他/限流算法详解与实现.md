
限流（Rate Limiting）是高并发系统“保命”的三板斧（缓存、降级、限流）之一。面试官不仅会问算法原理，更看重你是否懂它们的**优缺点、边界场景（如临界突发问题）以及代码落地**。

---

### 1. 核心结论 (Core Conclusion)

主流的单机限流算法有四种，按照演进顺序和平滑度依次是：
1. **固定窗口计数器（Fixed Window）**：最简单，但有“临界点突发”致命缺陷。
2. **滑动窗口（Sliding Window）**：解决了临界点问题，但内存占用较高。
3. **漏桶算法（Leaky Bucket）**：绝对平滑，强制按固定速率处理请求，**不允许任何突发流量**。
4. **令牌桶算法（Token Bucket）**：**工业界默认标配**（如 Guava `RateLimiter`），既能平滑限流，又**允许一定程度的突发流量**。

### 2. 深度解析与代码实现 (Detailed Breakdown & Implementations)

*(注：以下代码主要为了展示算法核心思想，真实高并发场景需考虑更严谨的锁或原子操作，分布式场景通常用 Redis + Lua 实现。)*

#### 2.1 固定窗口计数器算法 (Fixed Window)
* **原理：** 把时间划分为固定的片段（如 1 秒）。在这一秒内，每来一个请求计数器 +1。如果超过阈值就拒绝。下一秒计数器清零。
* **致命缺陷（临界点问题）：** 假设限流 100次/秒。0.99秒时来了 100 个请求，1.01秒时又来了 100 个请求。由于窗口在 1.00秒重置，这就导致在 0.99~1.01 这短短 0.02 秒内放过了 200 个请求，瞬间打垮系统！

```java
public class FixedWindowRateLimiter {
    private long windowSizeMs = 1000; // 窗口大小：1秒
    private int maxRequests = 100;    // 阈值：100次
    private AtomicInteger counter = new AtomicInteger(0);
    private volatile long windowStartTime = System.currentTimeMillis();

    public synchronized boolean tryAcquire() {
        long now = System.currentTimeMillis();
        // 进入下一个窗口，重置计数器
        if (now - windowStartTime >= windowSizeMs) {
            windowStartTime = now;
            counter.set(0);
        }
        // 尝试累加
        return counter.incrementAndGet() <= maxRequests;
    }
}
```

#### 2.2 滑动窗口算法 (Sliding Window)
* **原理：** 为了解决固定窗口的边界问题，将 1 秒的时间窗进一步划分为多个“小格子”（如 10 个 100ms 的格子）。随着时间推移，整个大窗口像履带一样向前滑动，丢弃旧格子，统计当前大窗口内所有小格子的总和。
* **优点：** 划分的格子越多，限流越平滑，完美解决临界点突发问题。（Sentinel 底层就是通过数组实现的滑动窗口 `LeapArray`）。

```java
public class SlidingWindowRateLimiter {
    private long windowSizeMs = 1000;
    private int maxRequests = 100;
    // 简单实现：用队列记录每次请求的精确时间戳
    private LinkedList<Long> requestTimestamps = new LinkedList<>();

    public synchronized boolean tryAcquire() {
        long now = System.currentTimeMillis();
        // 1. 踢出窗口外的过期请求
        while (!requestTimestamps.isEmpty() && now - requestTimestamps.peek() > windowSizeMs) {
            requestTimestamps.poll();
        }
        // 2. 判断当前窗口内的请求数是否超标
        if (requestTimestamps.size() < maxRequests) {
            requestTimestamps.add(now);
            return true;
        }
        return false; // 被限流
    }
}
```

#### 2.3 漏桶算法 (Leaky Bucket)
* **类比：** 想象一个底部开孔的漏斗。水（请求）以任意速度倒进漏斗（入水），但漏斗底部的孔只会以**固定的恒定速度**漏水（处理请求）。如果倒水太快漏斗满了，多余的水就会溢出（拒绝请求）。
* **特点：** 强行把原本突发的流量“整形”成了匀速的流量。适合保护数据库等绝对不能受冲击的下游系统。

```java
public class LeakyBucketRateLimiter {
    private long capacity = 100;       // 桶的容量（最大突发排队数）
    private double leakRate = 10.0;    // 漏水速率：每秒漏 10 个请求
    private double currentWater = 0;   // 当前桶里的水量
    private long lastLeakTime = System.currentTimeMillis();

    public synchronized boolean tryAcquire() {
        long now = System.currentTimeMillis();
        // 1. 计算过去这段时间漏掉了多少水
        double leakedWater = (now - lastLeakTime) / 1000.0 * leakRate;
        // 2. 更新桶里的当前水量（不能小于0）
        currentWater = Math.max(0, currentWater - leakedWater);
        lastLeakTime = now;

        // 3. 判断桶满没满
        if (currentWater < capacity) {
            currentWater++; // 进水
            return true;
        }
        return false; // 桶满了，溢出被限流
    }
}
```

#### 2.4 令牌桶算法 (Token Bucket) - ⭐️ 面试重点
* **类比：** 系统以恒定的速度向桶里发放“令牌”。请求进来时，必须从桶里拿走一个令牌才能被处理。如果桶空了，请求就被限流。
* **为什么它是王道？** 漏桶是严格匀速出水，而令牌桶允许你**“攒人品”**。如果系统之前一直很闲，桶里攒满了令牌，此时突然来了一波大流量，它们可以瞬间消耗掉所有令牌立刻被处理，从而**允许一定程度的突发流量**。这更符合互联网业务的真实诉求。

```java
public class TokenBucketRateLimiter {
    private long capacity = 100;        // 桶的容量：最多攒 100 个令牌
    private double tokenRate = 10.0;    // 发放速率：每秒 10 个令牌
    private double currentTokens = 100; // 当前令牌数（初始全满）
    private long lastRefillTime = System.currentTimeMillis();

    public synchronized boolean tryAcquire() {
        long now = System.currentTimeMillis();
        // 1. 动态计算这段时间应该补充多少令牌 (懒加载模式，不用单独起线程发令牌)
        double generatedTokens = (now - lastRefillTime) / 1000.0 * tokenRate;
        currentTokens = Math.min(capacity, currentTokens + generatedTokens);
        lastRefillTime = now;

        // 2. 尝试消费令牌
        if (currentTokens >= 1) {
            currentTokens--;
            return true;
        }
        return false; // 没令牌了，限流
    }
}
```

### 3. 面试官视角 (The Interviewer's Lens)

**高频追问与实战陷阱：**

1. **追问：漏桶和令牌桶有什么本质区别？**
 * *满分回答：* 漏桶限制的是**数据的流出速率**（绝对平滑，无视突发）；令牌桶限制的是**数据的平均流入速率**（允许一定程度的突发流量）。在 Web 应用中，保护自身通常用令牌桶（允许突发），调用脆弱的第三方 API 通常用漏桶（严格匀速）。
2. **追问：在真实的微服务集群中，单机限流够用吗？怎么做分布式限流？**
 * *满分回答：* 单机限流只保护当前节点。如果要限制某个用户对整个集群的访问频率，必须用分布式限流。业界最通用的做法是 **Redis + Lua 脚本**。比如用 Redis 维护一个基于滑动窗口的 ZSet，或者用 Lua 脚本原语执行令牌桶的 `(计算增量 -> 补充 -> 扣减)` 逻辑，保证这段操作的原子性。
3. **陷阱：Guava 的 `RateLimiter` 用的是什么算法？**
 * *满分回答：* 令牌桶算法。而且它支持两种模式：`SmoothBursty`（平滑突发，应对一般的突发流量）和 `SmoothWarmingUp`（平滑预热，系统冷启动时，发放令牌的速率会从慢慢慢变为正常，防止冷系统被瞬间打死）。这也是大厂高级面试常考的源码点。

---

### 4. 总结 (Summary)

* **固定/滑动窗口：** 侧重于“拒绝”，统计维度简单。滑动窗口（如 Sentinel）是单机防刷利器。
* **漏桶算法：** 侧重于“整形”，强行按固定频率处理，绝不允许突发（常用于网关层保护极其脆弱的遗留老系统）。
* **令牌桶算法：** 侧重于“弹性和平滑”，允许攒满令牌后的瞬间突发（Guava `RateLimiter` 默认实现，适用绝大部分互联网场景）。
