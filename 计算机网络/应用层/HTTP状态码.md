这个问题非常接地气！HTTP状态码是客户端和服务器沟通的“暗号”。在面试中，考官不仅要求你背出常见的数字，更会设置具体的生产环境故障（比如Nginx报错），让你通过状态码来定位问题。

我们依然按照\*\*“结论 -> 原理 -> 面试视角”\*\*来为你系统拆解。

---

### 💡 核心结论

HTTP 状态码由 **3位数字** 组成，第一位数字决定了响应的**五大核心类别**：

-   **1xx（信息）**：收到请求，正在处理。
    
-   **2xx（成功）**：事情顺利办成了！
    
-   **3xx（重定向）**：我这办不了，你去找别人（或用你之前的缓存）。
    
-   **4xx（客户端错误）**：你提交的申请表填错了，退回！
    
-   **5xx（服务端错误）**：我们的系统崩溃了，晚点再来！
    

---

### 🔍 深度原理解析 (Deep Dive)

面试中不需要把所有状态码都背下来，但以下几个是**必须掌握的高频考点**：

#### 2xx 成功类 (Success)

-   **200 OK**：最常见，请求成功，通常带有响应的 Body 数据。
    
-   **204 No Content**：请求成功，但**没有 Body 数据**返回。常用于前端发起的 `OPTIONS` 预检请求，或者仅仅用来上报数据不需要返回结果的 API。
    
-   **206 Partial Content**：**断点续传/分块下载**的核心。客户端只请求了文件的一部分（带有 `Range` 请求头），服务器成功返回了这一块数据。看视频拖拽进度条全是靠它。
    

#### 3xx 重定向类 (Redirection)

-   **301 Moved Permanently（永久重定向）**：这个网页已经永久搬家了。浏览器会自动把新的地址**缓存**下来，以后就算你输入老地址，浏览器也会直接跳到新地址。（有利于搜索引擎 SEO 权重转移）。
    
-   **302 Found（临时重定向）**：网页临时搬家。浏览器每次还是会先访问老地址，老地址再把你指引到新地址。（常用于大型活动的临时跳转）。
    
-   **304 Not Modified（未修改）**：**极其重要！缓存机制的核心**。服务器告诉客户端：“你要的数据自从上次给你之后就没变过，你直接用你本地的缓存吧！”此时响应报文**不包含 Body**，极大地节省了带宽。
    

#### 4xx 客户端错误类 (Client Error)

-   **400 Bad Request**：前端发来的请求参数有问题（比如格式不对、漏了必填项），后端解析失败。
    
-   **401 Unauthorized**：**未认证**。也就是“你是谁？”（通常是没带 Token 或者 Token 过期）。
    
-   **403 Forbidden**：**无权限**。也就是“我知道你是谁，但你没资格这么做。”（比如普通用户尝试删除超级管理员）。
    
-   **404 Not Found**：请求的资源（URL）在服务器上不存在。
    

#### 5xx 服务端错误类 (Server Error)

-   **500 Internal Server Error**：后端代码抛出了未捕获的异常（比如空指针 `NullPointerException`），纯纯的后端背锅。
    
-   **502 Bad Gateway（网关错误）**：通常是 Nginx 作为代理转发给后端 Tomcat/Go 服务时，发现\*\*后端服务挂了（进程死了）\*\*或者拒绝连接。
    
-   **503 Service Unavailable**：服务器由于负载过高或停机维护，暂时无法处理请求（通常会配合 `Retry-After` 告诉客户端等多久重试）。
    
-   **504 Gateway Timeout（网关超时）**：Nginx 连上了后端服务，但**后端服务处理得太慢了**（比如数据库死锁、慢查），Nginx 等不及了，直接断开并返回 504。

### 🧐 面试官视角 (The "Interviewer's Lens")

面试官最喜欢用生产环境的真实场景来考察你对 3xx 和 5xx 的理解。

#### 常见追问 (Follow-ups)

1.  **问：Nginx 代理后端服务时，502 和 504 有什么本质区别？（高频实战题 🌟）**
    

-   *答：* **502 (Bad Gateway)** 是连不上，说明后端进程根本不在了（比如 OOM 崩溃，或者没启动）；**504 (Gateway Timeout)** 是连上了但没回音，说明后端进程活着，但可能线程池满了、或者在等数据库慢查询，导致超时。
    

2.  **问：301 和 302 重定向分别会带来什么问题？**
    

-   *答：* 301 因为会被浏览器死死缓存，一旦后端想把规则改回来，如果用户不清理浏览器缓存是无法生效的，所以生产环境要**慎用 301**。302 有可能会被搜索引擎认为是作弊（URL 劫持），导致 SEO 降权。
    

#### 常见避坑 (Pitfalls)

-   **HTTP 状态码 vs 业务状态码混淆**：
    
-   *坑点：* 很多现在的业务系统，无论成功失败，HTTP 状态码全部返回 **200 OK**，然后在 JSON Body 里面写 `{"code": 500, "msg": "密码错误"}`。
    
-   *面试回答指南：* 面试时你可以客观分析——严格遵守 RESTful 规范的话，密码错误应该返回 4xx；但国内大部分互联网公司为了前端统一拦截、防止运营商劫持网络请求，更倾向于统一返回 200 HTTP 状态码，通过自定义的业务 Code 来表达错误。**展现出这种“规范 vs 工程实践”的辩证思维，是极大的加分项！**

### 💻 极客实践：用 Shell 命令体验 301 跳转

在排查域名重定向配置时，我们经常用 `curl -I`（只获取 Header，不下载 Body）来观察状态码。

```bash
# 我们用 http 去访问 github (github 强制要求 https)
curl -I http://github.com

# ----------------- 输出解析 -----------------
# 第一行清晰地看到了 301 永久重定向
HTTP/1.1 301 Moved Permanently
Content-Length: 0
# Location 字段告诉客户端：“新地址在这里，快去请求它！”
Location: https://github.com/
```

### 📝 一分钟速记总结 (Summary)

1.  **五大门派**：1xx待续、2xx成功、3xx跳转、4xx你错、5xx我错。
    
2.  **重定向双雄**：301 是永久搬家（浏览器会缓存），302 是临时出差。304 依靠缓存省流量。
    
3.  **网关双煞**：502 是后端死透了（连不上），504 是后端太慢了（等超时）。
