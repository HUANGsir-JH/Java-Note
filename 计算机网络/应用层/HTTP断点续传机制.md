这个问题非常实用！我们在用迅雷下载大文件、或者在 B站看长视频随意拖拽进度条时，背后全靠 **HTTP 断点续传（Resumable Download / Partial Content）** 机制。

在面试中，这道题不仅考察你对 HTTP 报文 Header 的熟悉程度，还能顺带考察你对**数据一致性（缓存与校验）以及多线程并发**的理解。

我们继续按照\*\*“结论 -> 原理 -> 面试视角”\*\*来为你系统拆解。

---

### 💡 核心结论

**HTTP 断点续传**是指客户端在下载（或上传）文件时，如果网络中断或暂停，下次恢复时**不需要从头开始，而是从上次断开的字节位置继续传输**。 它的核心依赖于三个关键点：

1.  客户端使用 `Range` 请求头告诉服务器要哪一段数据。
    
2.  服务器返回状态码 `206 Partial Content` 和 `Content-Range` 响应头。
    
3.  双方通过 `ETag` 或 `Last-Modified` 确保文件在暂停期间没有被偷偷篡改。
    

---

### 🔍 深度原理解析 (Deep Dive)

我们可以把下载大文件比作\*\*“抄写一本厚厚的书”\*\*：

#### 1\. 试探：服务器支持“书签”吗？ (Accept-Ranges)

当你第一次请求文件时（或者发个 HEAD 请求），服务器的响应头里如果带有 `Accept-Ranges: bytes`，就等于告诉客户端：“我支持按字节切分，你可以随便挑着读。” *(如果不支持，服务器会返回* `Accept-Ranges: none`*，你就只能老老实实从头抄到尾。)*

#### 2\. 核心：切片请求与响应 (Range & 206)

-   **客户端请求（夹书签）**：假设你已经下载了 1000 个字节，断网了。重连后，客户端发请求带上 Header：`Range: bytes=1000-`（意思是：老板，给我第 1000 个字节往后的所有内容）。
    
-   **服务端响应（给特定页码）**：服务器收到后，不再返回 `200 OK`，而是返回 `206 Partial Content`。同时在 Header 里带上 `Content-Range: bytes 1000-4999/5000`（意思是：你要的数据是 1000 到 4999 字节，这个文件总共有 5000 字节，拿去吧！）。
    

#### 3\. 校验：书还是原来那本书吗？ (If-Range)

-   **痛点**：如果你暂停下载了一星期，这期间服务器上的文件被更新了怎么办？如果直接拼上去，文件就损坏了！
    
-   **解法**：第一次下载时，服务器会给你一个文件的唯一标识 `ETag`（比如文件的 MD5 哈希值）或者最后修改时间 `Last-Modified`。
    
-   恢复下载时，客户端会带上 `If-Range: <原来的ETag>`。
    
-   服务器一对比：如果 ETag 没变，说明是同一本书，返回 `206` 和剩下的数据。
    
-   如果 ETag 变了，说明书被改版了，服务器会直接忽略 Range，返回 `200 OK` 和**全新的完整文件**。

### 🧐 面试官视角 (The "Interviewer's Lens")

面试官考察这道题，主要是想看你有没有**触类旁通**的工程思维。

#### 常见追问 (Follow-ups)

1.  **实战问法：“你知道迅雷、IDM 这类下载器，为什么下载速度比浏览器自带下载快那么多吗？”（高频实战题 🌟）**
    

-   *满分回答：* 因为它们利用了 HTTP 的 `Range` 机制实现了**多线程并发下载**。比如一个 100MB 的文件，下载器会开启 5 个线程，分别请求 `bytes=0-20M`, `20M-40M`, `40M-60M` 等等。这就相当于 5 个人同时帮你抄一本书的不同章节，最后在本地拼接成完整文件，把网络带宽“榨干”。
    

2.  **高级追问：“下载可以利用 HTTP 协议自带的 Range 实现断点续传，那上传大文件呢？”（拔高题 🌟）**
    

-   *满分回答：* HTTP 协议**原生并不完美支持**大文件断点上传。真实的工程实践中（比如网盘上传 10GB 电影），大文件断点上传通常是在\*\*应用层（业务代码）\*\*实现的：
    

1.  前端（JS）用 `File.slice()` 把大文件切成比如 5MB 一个的分片。
    
2.  前端用多个 HTTP POST 请求把这些分片（Chunk）并发传给后端，并带上分片序号（index）。
    
3.  后端全部收齐后，前端再发一个“合并（Merge）”请求，后端将所有碎片合并成完整文件。
    

#### 常见避坑 (Pitfalls)

-   *死记硬背 200 OK*：有些候选人认为只要成功拿到数据就是 `200 OK`。在断点续传场景下，**必须是** `206 Partial Content`。如果返回了 200，说明服务器不支持断点续传，把整个大文件又给你发了一遍。

### 💻 极客实践：用 curl 模拟多线程下载器

我们可以用命令行自己当一回“多线程下载器”，只请求文件的一部分：

```bash
# 假设有一个 1000 字节的 text.txt 文件，我们只要前 100 个字节
curl -i -H "Range: bytes=0-99" http://example.com/text.txt

# ----------------- 服务器返回报文解析 -----------------
HTTP/1.1 206 Partial Content
# 明确告诉你：这是 0到99字节，总大小是 1000
Content-Range: bytes 0-99/1000  
# 这个块的长度是 100 字节
Content-Length: 100             
# 文件的唯一身份证，用于 If-Range 校验
ETag: "33a64df551425fcc55e4d42a148795d9f25f89d4"

(这里是 \r\n 空行)
(这里是 text.txt 的前 100 个字节内容...)
```

### 📝 一分钟速记总结 (Summary)

1.  **核心头字段**：请求用 `Range` 画线，响应看 `Content-Range` 给范围。
    
2.  **核心状态码**：认准 `206 Partial Content`，返回 200 意味着重头再来。
    
3.  **数据一致性**：用 `If-Range` + `ETag` 确认文件没被偷偷修改。多线程并发下载也是基于此原理。
