这是分布式系统中最高频的面试题之一。

### 1\. 核心结论 (Core Conclusion)

在分布式环境下，**保证缓存与数据库的“强一致性”代价极高（性能剧降）**。绝大多数互联网大厂追求的是\*\*“最终一致性”\*\*。

目前业界最主流、性价比最高的方案是：**Cache Aside Pattern（旁路缓存模式）**，即：**先更新数据库，再删除缓存**，并配合 **Binlog 异步重试**。

---

### 2\. 深度方案拆解 (Detailed Breakdown)

#### ① 方案选型：为什么是“删除”而不是“更新”缓存？

-   **更新缓存的弊端**：
    

1.  **写多读少时浪费性能**：如果数据频繁更新但很少被读取，频繁更新缓存纯属浪费 CPU。
    
2.  **并发乱序风险**：线程 A 先更新 DB，线程 B 后更新 DB；由于网络抖动，线程 B 的缓存先更新，线程 A 的后更新，导致缓存里存的是旧值（A 的值）。
    

-   **删除缓存的好处**：操作简单，且下次读取时自动从数据库加载最新值，天然保证了数据闭环。
    

#### ② 方案选型：为什么先更新数据库，再删除缓存？

这是为了最大限度降低数据不一致的概率：

1.  **先删缓存，再更新 DB**：
    

-   *风险*：线程 A 删了缓存，还没来得及更新 DB。此时线程 B 进来读，发现缓存没了，去 DB 读了**旧值**并写入缓存。接着 A 更新了 DB。结果：**缓存里全是旧数据**。
    

2.  **先更新 DB，再删除缓存**：
    

-   *风险*：只有在“缓存刚好失效 + 线程 A 读 DB + 线程 B 写 DB + 线程 B 删缓存 + 线程 A 写缓存”这个极端顺序下才会出错。由于写 DB 通常比读 DB 慢得多，这种情况发生的概率极低。
    

#### ③ 进阶方案：如何解决“删除缓存失败”？（可靠性保证）

如果数据库更新成功，但删除缓存时网络波动失败了，依然会有脏数据。

-   **解决思路**：**异步重试机制**。
    

1.  **消息队列 (MQ)**：删除失败后，将该 Key 放入 MQ，由消费者不断重试直到成功。
    
2.  **订阅 Binlog (推荐)**：使用工具（如阿里巴巴的 **Canal**）模拟成 MySQL 的从节点，监听 Binlog。一旦数据库发生变动，Canal 异步通知程序去删除对应的缓存。这样把缓存过期逻辑从业务代码中解耦出来。
    

#### ④ 特殊场景：延时双删 (Delayed Double Delete)

如果你使用的是 **MySQL 读写分离架构**，主从同步会有延迟。

-   **痛点**：你删了缓存，更新了主库。但线程 B 读的是从库（还没同步完），又把旧数据塞回了缓存。
    
-   **对策**：先删缓存 -> 更新 DB -> **睡眠 500ms（等待主从同步完成）** -> 再次删除缓存。

### 3\. 面试官视角 (Interviewer's Lens)

#### 问：能不能用分布式锁来保证强一致性？

-   **答案**：可以，但没必要。使用分布式锁（如 Redisson 的读写锁 `RReadWriteLock`）可以让读写操作完全串行化。但这样 Redis 的高性能优势就荡然无存了。**除非是金融核心账务数据，否则不建议。**
    

#### 问：如果业务要求缓存必须和数据库实时一致，你怎么设计？

-   **答案**：
    

1.  考虑是否真的需要缓存？如果一致性要求高到不能忍受 100ms 的延迟，直接读数据库加索引可能是更好的选择。
    
2.  采用 **Write Through（穿透写）** 模式，由缓存服务统一封装数据库和缓存的操作，对外透明，内部加锁。

### 4\. 代码演示：Binlog 异步处理逻辑 (伪代码)

```java
// 1. 业务代码只管更新数据库
public void updateUser(User user) {
    userMapper.update(user); // 更新 DB
    // 不在这里删缓存，防止业务逻辑复杂导致缓存漏删或失败
}

// 2. 独立组件（如 Canal 监听器）异步处理
public void onBinlogEvent(Event event) {
    String key = "user:cache:" + event.getUserId();
    boolean success = redis.delete(key);
    
    if (!success) {
        sendToRetryQueue(key); // 进入 MQ 重试机制
    }
}
```

### 5\. 总结 (Summary)

1.  **主流选型**：先更新 DB，后删除缓存。
    
2.  **并发修复**：针对读写分离场景，使用“延时双删”。
    
3.  **可靠性增强**：利用 Canal 监听 Binlog 异步删除，配合 MQ 重试。
    
4.  **底层逻辑**：用性能换取最终一致性，而非强一致性。
    

**这就是缓存一致性的经典套路。顺着这个思路，面试官接下来的“杀招”通常是：如果缓存不幸被击穿或雪崩了，该怎么在离数据库最近的地方挡住流量？你想聊聊“布隆过滤器”还是“限流熔断”？**
