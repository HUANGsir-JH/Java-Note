设计秒杀系统是 Java 后端面试中的“压轴题”，它不仅考察你对 Redis、MQ 的掌握，更考察你对**高并发架构设计**的系统性思考。

---

### 1\. 核心结论 (Core Conclusion)

秒杀系统的核心思想是：**“层层过滤，削峰填谷，内存扣减，异步持久化”**。

-   **高并发**：请求不能直达数据库，必须挡在缓存层（Redis）。
    
-   **防止超卖**：利用 Redis 的单线程原子性（Lua 脚本）进行库存预扣减，而不是依赖数据库事务。
    

---

### 2\. 深度方案拆解 (Detailed Breakdown)

我们将秒杀系统分为四道防线：

#### 第一道防线：网关/前端层（限流与静态化）

1.  **资源静态化**：秒杀页面上的按钮、图片、CSS 全部上 CDN，不经过后端。
    
2.  **验证码/隐藏接口**：防止脚本自动抢购，点击后弹出验证码，或者通过动态 URL（URL 后面带随机 Hash）防止提前刷接口。
    
3.  **限流 (Rate Limiting)**：使用 Sentinel 或 Nginx 在网关层拦截。如果 1 万人抢 100 个商品，其实 90% 的请求在这一层就可以直接返回“已售罄”。
    

#### 第二道防线：缓存层（核心扣减逻辑 - 防超卖）

**严禁在数据库里** `update stock = stock - 1`**！**

1.  **库存预热**：秒杀开始前，将库存数量提前加载到 Redis。
    
2.  **原子扣减 (Lua 脚本)**：
    

-   使用 Lua 脚本保证“查询库存 + 扣减库存”的原子性。
    
-   逻辑：`if get(stock) > 0 then decr(stock) return 1 else return 0 end`。
    
-   **为什么用 Lua？** 它可以防止多个请求同时看到剩余 1 件库存，从而导致多扣的情况。
    

#### 第三道防线：异步层（消息队列削峰）

1.  **异步下单**：Redis 扣减成功后，立即返回给用户“排队中/抢购成功”，然后向消息队列（MQ）发送一条下单消息。
    
2.  **流量削峰**：数据库（MySQL）每秒只能抗几千 QPS，而 MQ 可以起到缓冲作用，订单服务按照自己的节奏从 MQ 消费消息，慢慢写库。
    

#### 第四道防线：数据库层（最终兜底）

1.  **唯一索引**：用户 ID + 商品 ID 设置联合唯一索引，防止同一个用户抢两次。
    
2.  **乐观锁/版本号**：更新时带上 `where stock > 0`，作为物理上的最后一道防线。

### 3\. 面试官视角 (Interviewer's Lens)

#### 问：如何防止同一个用户多次点击抢购？

-   **答案**：
    

1.  **前端**：点击后按钮置灰。
    
2.  **后端**：利用 Redis 的 `SETNX` 设置一个短时间内的业务分布式锁（例如：`lock:user_id:item_id`，有效期 10 秒）。
    

#### 问：Redis 扣减了库存，但用户最后没付款，库存不就白扣了吗？

-   **答案**：**库存回滚机制**。
    

1.  订单在创建后会进入一个定时任务或延迟队列（如 RocketMQ 的延迟消息）。
    
2.  如果 15 分钟未支付，系统自动取消订单，并在 Redis 中执行 `INCR` 将库存加回去。
    

#### 问：如果 Redis 挂了，秒杀是不是彻底凉了？

-   **答案**：秒杀通常采用 **Redis 集群/主从架构**。对于极端重要的活动，可以采用“多机房多实例”部署。如果 Redis 彻底不可用，应立即触发**服务熔断**，保护数据库不被打死，此时用户会看到“系统繁忙”。

### 4\. 代码演示：Lua 脚本原子扣减库存 (Java)

```java
// Redis Lua 脚本：判断并扣减库存
String luaScript = 
    "local stock = tonumber(redis.call('get', KEYS[1]));" +
    "if (stock <= 0) then return -1; end;" + // 库存不足
    "redis.call('decr', KEYS[1]);" +
    "return stock - 1;"; // 返回剩余库存

// 执行脚本
Long result = redisTemplate.execute(
    new DefaultRedisScript<>(luaScript, Long.class),
    Collections.singletonList("seckill:stock:item_1001")
);

if (result >= 0) {
    // 扣减成功，发消息给 MQ
    mqProducer.sendOrderMessage(userId, itemId);
    return "抢购成功，正在为您生成订单";
} else {
    return "手慢了，商品已售罄";
}
```

### 5\. 总结 (Summary)

1.  **抗高并发**：请求拦截在 Redis 层，不要让流量压到数据库。
    
2.  **防超卖**：利用 **Redis Lua 脚本** 进行原子扣减，结合 **数据库唯一索引** 兜底。
    
3.  **用户体验**：**异步下单**，快速响应用户，剩下的脏活累活交给 MQ 慢慢做。
    

**秒杀系统是典型的“空间换时间”和“异步化”思维的结合。在解决了高并发扣库存后，面试官可能会深挖：如果数据库写订单太慢，导致 MQ 堆积怎么办？或者如何设计一个“分布式事务”来确保 Redis 扣减和订单创建的最终一致性？你想聊聊哪一个？**
