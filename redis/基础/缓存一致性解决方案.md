缓存一致性（Cache Consistency）是分布式系统中的经典难题。核心矛盾在于：**数据库（DB）和 Redis 是两个独立的组件，无法做到物理上的原子性操作**（除非引入极其复杂的分布式事务，但这会严重影响性能）。

在更新数据时，我们面临两个关键选择：

1.  **是更新缓存（Update）还是删除缓存（Delete）？**
    
2.  **是先操作数据库还是先操作缓存？**
    

---

### 一、 更新缓存 vs 删除缓存

在实际开发中，**绝大多数场景建议选择“删除缓存”**。

#### 1\. 更新缓存 (Update Cache)

-   **做法：** 每次数据库更新后，同步把新值写回 Redis。
    
-   **缺点：**
    
-   **计算代价高：** 如果缓存的值不是直接从数据库读出来的，而是经过复杂计算（如多表关联、聚合）得到的，频繁更新缓存会浪费性能。
    
-   **产生脏数据（并发竞争）：** 假设有两个线程 A 和 B 同时更新同一条数据：
    

1.  线程 A 更新数据库。
    
2.  线程 B 更新数据库。
    
3.  线程 B 更新缓存。
    
4.  线程 A 更新缓存。
    

-   **结果：** 数据库里是 B 的值，缓存里是 A 的值。**数据不一致。**
    

#### 2\. 删除缓存 (Delete Cache) —— 推荐做法

-   **做法：** 每次数据库更新后，直接把对应的 Redis Key 删掉。下次读取时发现 Miss，再从数据库加载最新的值。
    
-   **优点：**
    
-   **操作简单：** 无论计算多复杂，删掉即可。
    
-   **避免并发冲突：** 删除了之后，后续的读请求自然会触发同步加载。
    

---

### 二、 先操作 DB 还是先操作缓存？

即使决定了“删除缓存”，操作顺序依然决定了系统的鲁棒性。

#### 方案 A：先删缓存，再更数据库

-   **风险：** 在高并发下极易出现不一致。
    
-   **不一致场景：**
    

1.  线程 A 删除缓存。
    
2.  线程 B 请求数据，发现缓存不存在（Miss）。
    
3.  线程 B 去数据库读出**旧值**。
    
4.  线程 B 将**旧值**写入缓存。
    
5.  线程 A 将**新值**写入数据库。
    

-   **结果：** 缓存中永远是旧值，直到下次更新或过期。
    
-   **解决：延时双删（Delayed Double Delete）**
    
-   先删缓存 -> 更新数据库 -> **睡眠几百毫秒** -> 再次删除缓存。
    
-   *注：睡眠时间是为了确保读请求产生的脏数据已经写入缓存，然后再把它删掉。但这很难评估精准的睡眠时间。*
    

#### 方案 B：先更数据库，再删缓存（Cache Aside Pattern 常用方案）

-   **风险：** 如果第二步“删除缓存”失败了，会导致缓存里是旧值。
    
-   **不一致场景（概率极低）：**
    

1.  缓存刚好失效。
    
2.  线程 A 读数据库，得到旧值。
    
3.  线程 B 更新数据库，并删除缓存。
    
4.  线程 A 将旧值写入缓存。
    

-   *注：这种情况发生的条件非常苛刻，要求读操作比写操作还慢，实际生产中很少见。*
    

---

### 三、 如何保证“删除缓存”一定成功？（进阶方案）

在“先更数据库，再删缓存”的策略中，如果第二步删除失败了怎么办？

#### 1\. 消息队列重试机制

1.  更新数据库。
    
2.  删除缓存失败。
    
3.  将“删除失败的 Key”发送到消息队列（MQ）。
    
4.  消费端不断尝试删除，直到成功。
    

#### 2\. 订阅数据库 Binlog（最优雅的方案）

1.  应用程序只管更新数据库。
    
2.  使用工具（如 **Canal**）模拟成数据库的从节点，监听 MySQL 的 Binlog 变更。
    
3.  专门的服务解析 Binlog，提取 Key，并在 Redis 中执行删除操作。
    

-   **优点：** 业务代码解耦，保证最终一致性。
    

---

### 四、 总结与建议

1.  **一般场景：** 采用 **Cache Aside Pattern**（先更新数据库，再删除缓存）。设置合理的 **TTL（过期时间）** 作为兜底方案，即使出现短暂不一致，缓存到期后也会自动修复。
    
2.  **高并发且一致性要求高：** 采用 **先更新数据库 + 订阅 Binlog 异步删除缓存**。
    
3.  **强一致性场景：** 如果要求缓存和数据库必须实时一致，Redis 可能不适合。应当考虑 **读写锁（ReadWriteLock）**，或者直接读数据库。
    

**金句总结：**

-   **更新**缓存不如**删除**缓存。
    
-   **先删**缓存不如**后删**缓存。
    
-   **最终一致性**是缓存架构的基石，不要过分追求**强一致性**（除非愿意牺牲巨大的性能）。
