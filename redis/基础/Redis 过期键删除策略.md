Redis 的过期删除策略是回答“内存去哪儿了”和“为什么过期键还在”的关键。

---

### 1\. 核心结论 (Core Conclusion)

Redis 采用的是 **“惰性删除 (Lazy Expiration)”** 与 **“定期删除 (Periodic Expiration)”** 配合使用的折中方案。

-   **惰性删除：** 节省 CPU，浪费内存。
    
-   **定期删除：** 浪费 CPU，节省内存。 两者互补，达到了性能与内存占用之间的完美平衡。
    

---

### 2\. 深度原理剖析 (Detailed Breakdown)

#### ① 惰性删除 (Lazy Expiration)

-   **逻辑：** 当客户端访问一个 Key 时，Redis 会先检查这个 Key 是否过期。如果过期了，就立即删除并返回 `nil`；如果没有过期，才正常返回。
    
-   **优点：** 对 CPU 极其友好。只有在处理请求时才顺便处理过期，不会在后台额外占用宝贵的 CPU 资源。
    
-   **缺点：** 对内存不友好。如果一大批 Key 过期了，但此后没有任何人访问它们，它们会一直霸占着内存，造成“内存泄漏”。
    

#### ② 定期删除 (Periodic Expiration)

-   **逻辑：** Redis 会将设置了过期时间的 Key 放入一个独立的字典中。每隔一段时间（默认 100ms），Redis 就会随机抽取一批 Key 进行检查。
    
-   **执行流程：**
    

1.  从过期字典中随机抽取 20 个 Key。
    
2.  删除其中已过期的 Key。
    
3.  如果过期的 Key 占比超过 25%，则重复步骤 1（循环触发）。
    

-   **限制：** 为了防止这个循环扫描导致主线程卡死，Redis 设置了严格的**执行时间上限**（默认 25ms）。
    
-   **优点：** 通过限制频率和时长，减少了对 CPU 的影响，同时能有效清理不再被访问的过期键。

### 3\. 为什么不使用“定时删除”？ (The Rejected Alternative)

**面试官可能会问：** 为什么不在设置过期时间时，直接起一个定时器，时间一到就删？

-   **答案：** **太耗 CPU 了。** 如果有 100 万个 Key 同时过期，Redis 就需要维护 100 万个定时器。在高性能的 I/O 环境下，管理定时器的时间复杂度开销是无法接受的。
    

---

### 4\. 特殊场景下的表现 (Edge Cases)

#### ① RDB 和 AOF 如何处理过期键？

-   **RDB 生成：** 生成快照时，已过期的 Key 不会被写入 RDB 文件。
    
-   **RDB 载入：** 载入时会检查，过期的 Key 会被忽略。
    
-   **AOF 写入：** Key 过期后，Redis 会在 AOF 文件末尾追加一条 `DEL` 命令。
    
-   **AOF 重写：** 重写时，已过期的 Key 不会进入新的 AOF 文件。
    

#### ② 主从同步中的坑

-   **核心逻辑：** **从节点不具备主动删除过期键的能力。**
    
-   **流程：** 当主节点的 Key 过期时，主节点会模拟一条 `DEL` 命令发给从节点。
    
-   **后果：** 如果主从同步有延迟，或者是旧版本的 Redis（3.2 以前），可能会出现“在从节点能查到主节点已过期数据”的情况。

### 5\. 面试官视角 (Interviewer's Lens)

#### 问：过期删除 (Expiration) 和 内存淘汰 (Eviction) 有什么区别？

这是最经典的陷阱题。

-   **过期删除：** 针对设置了 `TTL` 的 Key，时间到了就该死。
    
-   **内存淘汰：** 针对内存写满了的情况，根据 LRU/LFU 强制踢走一部分 Key（不管你过没过期），是为了保命（保服务可用）。
    

#### 问：如果定期删除和惰性删除都没删掉，内存还是满了怎么办？

-   *答案：* 这时候才会触发我们上一轮聊到的 **“内存淘汰策略”**（如 `allkeys-lru`）。
    

#### 潜在陷阱 (Pitfalls)：

-   **BigKey 删除阻塞：** 如果一个过期的 Key 是包含几百万个元素的 Hash，无论是定期还是惰性删除，都会导致主线程在 `DEL` 时卡顿。
    
-   **优化建议：** 开启 `lazyfree-lazy-expire yes`，让过期删除在后台线程异步执行（`UNLINK` 而不是 `DEL`）。
    

---

### 6\. 总结 (Summary)

1.  **惰性删除：** 访问时才删，不查不删。
    
2.  **定期删除：** 每秒 10 次随机抽查，限时执行。
    
3.  **终极防护：** 如果前两者不给力，内存满了会触发“内存淘汰策略”。
