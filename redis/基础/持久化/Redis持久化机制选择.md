Redis 的持久化是保障数据不丢的核心能力，也是面试高频考点。面试官不仅想知道“是什么”，更想看你是否理解“为什么这么设计”和“在生产中怎么选”。

### 1\. 核心结论 (Core Conclusion)

Redis 提供两种持久化方式： ✅ **RDB（快照）** —— **“定时拍照”**，适合备份与恢复。 ✅ **AOF（追加日志）** —— **“逐条记账”**，适合高可靠性场景。

**选择原则：**

> **追求性能和恢复速度 → 用 RDB**
> 
> **追求数据安全 → 用 AOF**
> 
> **生产推荐：RDB + AOF 双开** —— 兼顾速度与安全。

### 2\. 深度原理剖析 (Detailed Breakdown)

#### ① RDB（Redis Database）—— 快照模式

##### ✅ 原理

-   定时或手动触发，将**当前内存中的所有数据**以二进制格式写入一个 `.rdb` 文件。
    
-   使用 `fork()` 创建子进程完成写盘，主线程继续服务，**不影响性能**。
    
-   文件紧凑，是 Redis 数据的完整“快照”。
    

##### ✅ 优点

| 优点  | 说明  |
| --- | --- |
| **恢复速度快** | 加载一个二进制快照文件，直接重建内存结构，比逐条执行命令快 10 倍以上。 |
| **文件小** | 压缩存储，占用磁盘空间少。适合备份、传输、冷备。 |
| **性能影响小** | 子进程异步写盘，主线程无阻塞（除非 fork 时内存过大）。 |
| **适合全量备份** | 适合做定时备份、灾备、主从同步初始数据。 |

##### ❌ 缺点

| 缺点  | 说明  |
| --- | --- |
| **可能丢数据** | 默认每 5 分钟快照一次，若 Redis 崩溃，最多丢失 5 分钟数据（可配置更频繁，但增加开销）。 |
| **大内存时 fork 阻塞** | 如果内存 > 10GB，`fork()` 会阻塞主线程几十到几百毫秒（Linux 写时复制机制下，子进程会复制页表，造成短暂卡顿）。 |
| **不支持增量** | 每次都是全量，无法记录“改了哪几条”。 |

##### 📌 典型配置

```conf
save 900 1     # 900秒内至少1个key变化 → 触发快照
save 300 10    # 300秒内至少10个key变化
save 60 10000  # 60秒内至少10000个key变化
```

---

#### ② AOF（Append Only File）—— 命令日志模式

##### ✅ 原理

-   记录**每一个写操作命令**（如 `SET key value`）到一个 `.aof` 文件。
    
-   类似 MySQL 的 binlog，重启时通过**重放日志**重建数据。
    
-   支持三种刷盘策略（`appendfsync`）：
    
-   `always`：每个命令都写入磁盘 → 最安全，最慢。
    
-   `everysec`（默认）：每秒刷一次 → 安全性与性能平衡。
    
-   `no`：由操作系统决定 → 最快，最危险。
    

##### ✅ 优点

| 优点  | 说明  |
| --- | --- |
| **数据更安全** | 默认每秒刷盘，最多丢失 1 秒数据，远优于 RDB。 |
| **可读性强** | AOF 文件是纯文本，可手动编辑修复（如误删了 key）。 |
| **支持重写** | 通过 `BGREWRITEAOF` 自动压缩日志，删除冗余命令（如连续 100 次 `INCR` → 变成 `SET key 100`）。 |

##### ❌ 缺点

| 缺点  | 说明  |
| --- | --- |
| **文件大** | 记录所有命令，文件体积远大于 RDB（可通过重写缓解）。 |
| **恢复慢** | 重放几千条命令比加载一个二进制快照慢得多。 |
| **写入压力大** | 若 `appendfsync always`，每条命令都同步磁盘，I/O 压力极高。 |
| **重写期间内存翻倍** | 重写时，子进程会生成新日志，此时内存中同时存在旧日志 + 新日志，可能触发 OOM。 |

##### 📌 典型配置

```conf
appendonly yes
appendfilename "appendonly.aof"
appendfsync everysec
auto-aof-rewrite-percentage 100
auto-aof-rewrite-min-size 64mb
```

### 3\. 面试官视角 (Interviewer's Lens)

#### 常见追问：

1.  **RDB 和 AOF 同时开启，重启时用哪个？**
    

-   *答案：* **优先使用 AOF**。因为 AOF 数据更完整，Redis 会自动选择 AOF 恢复。
    

2.  **AOF 重写是怎么实现的？会不会影响服务？**
    

-   *答案：* 通过 `fork()` 创建子进程，子进程读取当前内存快照，生成新日志；主线程继续写旧日志。重写完成后，用新文件替换旧文件。**不影响服务**，但会短暂增加内存和 CPU 压力。
    

3.  **如果 AOF 文件损坏了怎么办？**
    

-   *答案：* 用 `redis-check-aof --fix appendonly.aof` 修复。这是 AOF 的一大优势 —— 可人工干预。
    

#### 潜在陷阱 (Pitfalls)：

-   **不要说“AOF 比 RDB 更快”** —— AOF 是**写入快**，但**恢复慢**。
    
-   **不要忽略 fork 阻塞** —— 在大内存机器上，RDB 的 fork 可能造成服务抖动，这是生产环境的隐形杀手。
    
-   **不要只用 AOF 不用 RDB** —— AOF 文件太大时，恢复可能需要几分钟，而 RDB 只需几秒。

### 4\. 性能与安全对比表

| 维度  | RDB | AOF |
| --- | --- | --- |
| **数据安全性** | ⚠️ 低（最多丢几分钟） | ✅ 高（最多丢 1 秒） |
| **恢复速度** | ✅ 极快 | ⚠️ 较慢 |
| **文件大小** | ✅ 小 | ⚠️ 大（可重写） |
| **写入性能** | ✅ 高（异步） | ⚠️ 中（取决于 sync 策略） |
| **是否支持增量** | ❌ 否 | ✅ 是 |
| **是否可人工编辑** | ❌ 否 | ✅ 是 |
| **适用场景** | 备份、缓存、快速重启 | 金融、订单、强一致性 |

---

### 5\. 总结 (Summary)

-   **RDB = 快照备份** → 适合**恢复快、空间小、定期备份**。
    
-   **AOF = 命令日志** → 适合**数据安全、可恢复、可调试**。
    
-   **生产黄金组合：RDB + AOF 同时开启！**
    
-   用 RDB 做**全量备份**和**快速重启**；
    
-   用 AOF 做**增量保障**，确保数据不丢；
    
-   重启时 Redis 自动选 AOF，安全无忧。
