如果把单机 Redis 比作一家“个体路边摊”，那么 Redis 集群就是“连锁品牌”或者“大型超市”。

---

### 1\. 核心结论 (Core Conclusion)

**Redis 集群**是将数据分布在多个 Redis 节点上的分布式架构方案。 **为什么需要？** 主要是为了突破单机在 **存储容量（内存限制）**、**计算能力（并发瓶颈）** 和 **稳定性（单点故障）** 这三个维度的天花板。

---

### 2\. 三种模式详解 (Modes Breakdown)

Redis 的集群演进经历了三个阶段，分别对应不同的业务需求：

#### ① 主从复制 (Master-Slave) —— “数据备份”

-   **架构：** 一个 Master 节点，多个 Slave 节点。数据从 Master 同步到 Slave。
    
-   **特点：** 读写分离（Master 写，Slave 读）。
    
-   **优点：** 实现了数据备份，分担了读压力。
    
-   **缺点：** **没有自动故障转移**。如果 Master 挂了，需要人工把 Slave 提拔为 Master，且 Master 的内存容量依然是整个系统的上限。
    

#### ② 哨兵模式 (Sentinel) —— “自动化高可用”

-   **架构：** 在主从复制的基础上，额外增加了“哨兵”进程。
    
-   **职责：** 监控（Monitoring）、通知（Notification）、**自动故障转移（Automatic Failover）**。
    
-   **优点：** 真正实现了高可用。Master 挂了，哨兵会自动投票选出新 Master。
    
-   **缺点：** **依然没有解决存储上限问题**。所有节点存的数据都是一模一样的全量数据。
    

#### ③ Redis Cluster (官方集群模式) —— “横向扩展的终极方案”

-   **架构：** 真正的分布式去中心化架构，数据分布在多个 Master 上。
    
-   **核心机制：哈希槽 (Hash Slot)**。Redis Cluster 预设了 **16384** 个槽位，每个 Master 负责一部分槽位。
    
-   **优点：**
    

1.  **横向扩展：** 内存不够了？加机器就行，数据会自动迁移。
    
2.  **高性能：** 客户端直接连接集群中的任意节点，节点会告知数据在哪个槽。
    
3.  **内置高可用：** 每个 Master 都可以配自己的 Slave，Master 挂了，集群内部自动完成主从切换。
    

---

### 3\. 深度原理：为什么是 16384 个槽？ (Interviewer's Favorite)

这是面试中非常容易被问到的“刁难”题：

1.  **解耦：** 槽位（Slot）是逻辑层，节点（Node）是物理层。我们不是把 Key 直接映射到 Node，而是 `Key -> Slot -> Node`。这样在增加或减少机器时，只需要移动“槽”，而不需要重新计算所有数据的哈希值。
    
2.  **为什么不是 65535？** 因为 Redis 节点间通信会发送心跳包，心跳包里包含该节点负责的槽位信息。如果槽位太多（65k），心跳包头会变得很大（8kb），浪费带宽；而 16k 既能满足绝大多数大集群需求，包头也只有 2kb。

### 4\. 面试官视角 (Interviewer's Lens)

#### 问：Redis Cluster 是强一致性的吗？

-   **答案：** **不是。** Redis 集群为了追求高性能，采用的是 **异步复制**。在主从切换过程中，可能会丢失少量未同步到 Slave 的数据。
    

#### 问：什么是“脑裂”问题？

-   **场景：** 哨兵模式下，Master 因为网络波动和哨兵失联，哨兵选了新 Master。但旧 Master 其实没挂，还在接收客户端请求。
    
-   **后果：** 系统中同时出现了两个 Master。当网络恢复，旧 Master 降级为 Slave，它之前收到的新数据会被清空，导致数据丢失。
    
-   **对策：** 配置 `min-slaves-to-write`（要求 Master 必须至少有 N 个从节点在线且延迟在规定时间内，否则拒绝写入）。
    

#### 潜在陷阱 (Pitfalls)：

-   **批量操作失效：** 在 Cluster 模式下，如果 `MGET` 里的 Key 分布在不同的 Slot（即不同的物理节点），会直接报错。
    
-   *解决：* 使用 **Hash Tag**。例如 `SET {user1}:name` 和 `SET {user1}:age`，Redis 只会对花括号里的内容算哈希，确保它们落在同一个槽。

### 5\. 总结 (Summary)

| 模式  | 解决的问题 | 缺点  |
| --- | --- | --- |
| **主从** | 数据备份、读压力 | 无法自动恢复，内存受限 |
| **哨兵** | 自动故障转移 (HA) | 内存受限，配置复杂 |
| **Cluster** | **横向扩展 (容量)** + 高可用 | 客户端实现复杂，不支持跨节点批量操作 |
