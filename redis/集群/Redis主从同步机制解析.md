
Redis 的主从同步（Replication）是 Redis 高可用架构的基石。无论是哨兵模式还是集群模式，底层都依赖于这套机制。

---

### 1. 核心结论 (Core Conclusion)

Redis 主从同步分为两个维度：**全量同步（Full Resynchronization）** 和 **增量同步（Partial Resynchronization）**。
* **全量同步：** 用于从节点第一次连接主节点，或者主节点缓冲区失效的情况，通过 **RDB 文件** 进行同步。
* **增量同步：** 用于网络闪断恢复后的数据补齐，通过 **指令偏移量（Offset）** 在积压缓冲区中查找缺失数据。

---

### 2. 深度流程拆解 (Detailed Breakdown)

同步过程主要通过 `psync` 命令触发，分为三个关键阶段：

#### 第一阶段：建立连接与全量同步 (The First Time)
1. **发出请求：** 从节点发送 `psync {runID} {offset}`。第一次同步时，`runID` 为 `?`，`offset` 为 `-1`。
2. **生成快照：** 主节点执行 `bgsave` 生成 RDB 文件，同时将**此时开始**接收到的所有写命令缓存到 `replication buffer`（复制缓冲区）中。
3. **发送 RDB：** 主节点将 RDB 发送给从节点。从节点清空旧数据，载入新 RDB。
4. **同步增量：** RDB 发送完毕后，主节点将 `replication buffer` 中的写命令发给从节点执行。

#### 第二阶段：命令传播 (Normal State)
* 全量同步完成后，主从之间维护一个 **长连接**。
* 主节点每执行一个写命令，就会异步发送给从节点，保证数据实时一致。

#### 第三阶段：断线重连后的增量同步 (Recovery)
当网络短暂中断后，Redis 2.8+ 不再盲目进行全量同步，而是尝试“增量补票”：
1. **检查 Offset：** 主从节点各维护一个 `offset`。
2. **查找积压缓冲区：** 主节点内存中有一个固定大小的环形队列 `repl_backlog_buffer`（复制积压缓冲区）。
3. **判断逻辑：** 如果从节点丢失的 `offset` 还在这个环形队列里，主节点只发送缺失的部分。如果断开时间太长，缺失的数据被覆盖了，则无奈退化为**全量同步**。

---

### 3. 三个核心概念 (The Three Pillars)

1. **RunID：** 每个 Redis 实例启动时生成的唯一 ID。从节点根据它判断主节点是否变过（如果主节点重启了，RunID 变了，必须全量同步）。
2. **Offset (偏移量)：** 记录同步到了哪个位置。
3. **Replication Backlog Buffer：** 主节点上的环形缓冲区。它是**所有从节点共享**的。

---

### 4. 面试官视角 (Interviewer's Lens)

#### 问：主从同步会阻塞主线程吗？
* **答案：** **生成 RDB 的过程不阻塞**（fork 子进程），但 **Fork 瞬间**和**大 RDB 文件传输时的网络带宽占用**可能会影响性能。

#### 问：如果从节点太多，主节点会压力过大吗？
* **对策：** 使用 **“主 -> 从 -> 从” 的级联结构**（分层复制）。让部分从节点作为其他从节点的主，分担主节点的 RDB 生成和网络发送压力。

#### 问：主从数据不一致怎么办？
* **原因：** Redis 是异步同步。
* **建议：**
 * 监控 `master_repl_offset` 和 `slave_repl_offset` 的差值。
 * 配置 `min-slaves-max-lag`，如果从节点延迟太大，主节点拒绝写入。

#### 潜在陷阱 (Pitfalls)：
* **过期 Key 的问题：** 从节点不会主动删除过期 Key。它会等待主节点的 `DEL` 指令。这可能导致从节点在短时间内读到已过期的数据。
* **缓冲区溢出：** 如果 `replication buffer`（发送缓冲区）设置太小，而主节点写操作极快，会导致同步连接断开，进而陷入“全量同步 -> 失败 -> 再次全量同步”的死循环。

---

### 5. 总结 (Summary)

1. **初次见面看全量：** RDB 传全量，Buffer 传增量。
2. **网络闪断看偏移：** 只要缓冲区没满，Offset 补齐即可。
3. **异步同步快为先：** 追求高性能，不保证强一致。

**主从同步是 Redis 高可用的基础，那么如果主节点真的彻底“挂了”，该由谁来负责发现并把从节点提拔为主节点呢？这就是我们之前提到的“哨兵模式”或“集群模式”的职责了。你想深入聊聊哨兵是如何“投票”选主的吗？**
