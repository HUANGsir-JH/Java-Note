
这是一个非常经典且高频的面试题。理解 `HashMap` 的扩容机制，不仅能体现你对数据结构的掌握，还能展现你对位运算和性能优化的理解。

---

### 1. 核心结论 (Conclusion First)

`HashMap` 的扩容（Resize）是一个**容量翻倍**的过程。
* **触发条件：** 当 `size > threshold`（当前元素个数超过 阈值 = 容量 × 负载因子）时触发。
* **核心逻辑：** 创建一个容量为原先 **2 倍** 的新数组，将原数组中的元素重新计算索引并迁移到新数组中。
* **Java 8 优化：** 不再像 Java 7 那样重新计算每个 key 的 hash 值，而是通过**位运算**快速确定元素在新数组中的位置。

---

### 2. 深度解析 (Detailed Breakdown)

#### ① 触发阈值与容量计算
* **默认参数：** 默认初始容量 `16`，默认负载因子 `0.75`。
* **阈值计算：** `16 * 0.75 = 12`。当存入第 13 个键值对时，就会触发扩容。
* **为什么是 0.75？** 这是一个权衡值。如果太小（如 0.5），空间浪费严重，频繁扩容；如果太大（如 1.0），哈希冲突会非常剧烈，查询效率（时间复杂度）会下降。

#### ② 容量为什么必须是 2 的幂？
这是 `HashMap` 性能优化的核心。
* **索引计算公式：** `index = hash % length`。
* **位运算优化：** 当 `length` 是 2 的幂时，`hash % length` 等价于 `hash & (length - 1)`。**位运算的效率远高于取模运算。**
* **分布均匀：** 2 的幂减 1 后，二进制位全为 1（如 15 是 `1111`），能保证 `hash` 值的每一位都参与运算，减少冲突。

#### ③ Java 8 的“高低位”迁移逻辑（重点）
在 Java 8 中，迁移不再需要调用 `hash()` 函数重新计算位置，而是根据 `(e.hash & oldCap)` 的结果分为两组：
1. **结果为 0：** 元素留在原位置（`index`）。
2. **结果为 1：** 元素移动到新位置（`index + oldCap`）。

**类比：** 就像搬家，原来住在 5 号房的人，扩容后要么还住 5 号，要么住 `5 + 16 = 21` 号，非常规律。

### 3. 面试官视角 (The "Interviewer's Lens")

**追问 1：扩容是线程安全的吗？**
* **坑：** `HashMap` 在多线程下是不安全的。
* **风险：**
 * **Java 7：** 采用头插法迁移，多线程扩容可能导致**环形链表**，造成死循环（CPU 100%）。
 * **Java 8：** 改用尾插法，解决了死循环问题，但在并发下仍可能发生**数据覆盖/丢失**。
 * **对策：** 并发场景请使用 `ConcurrentHashMap`。

**追问 2：如果我知道要存 1000 个数据，初始化 `HashMap` 时给多少容量合适？**
* **公式：** `capacity = (expectedSize / loadFactor) + 1`。
* **建议：** 建议设置为 `2048`（或使用 `1000/0.75 + 1 = 1334` 之后向上取最近的 2 的幂）。这可以**避免在写入过程中发生多次扩容**，极大提升性能。

**追问 3：扩容时，红黑树怎么处理？**
* 如果桶中是红黑树，扩容时会拆分树。如果拆分后的子树节点数 **≤ 6**，红黑树会退化回链表。

---

### 4. 代码片段（原理模拟） (Code Snippet)

这是 Java 8 中确定新位置的核心逻辑伪代码：

```java
// e.hash 是元素的哈希值，oldCap 是扩容前的容量（如16）
if ((e.hash & oldCap) == 0) {
    // 元素在新数组中的索引 = 原索引
    newTab[j] = e; 
} else {
    // 元素在新数组中的索引 = 原索引 + oldCap
    newTab[j + oldCap] = e;
}
```

---

### 5. 核心总结 (Summary)

1. **倍增机制：** 扩容是容量翻倍（`oldCap << 1`），阈值随之翻倍。
2. **位算之美：** 利用 `n-1` 的掩码特性，通过 `&` 运算代替 `%` 运算，极大地提高了寻址和迁移效率。
3. **避坑指南：** 尽量预估数据量以减少重绘（Resize），并严禁在多线程环境下直接使用 `HashMap`。
