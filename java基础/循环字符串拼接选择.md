在 Java 开发中，字符串处理是最高频的操作之一，如何高效地进行字符串拼接是衡量一个开发者对 JVM 内存管理理解程度的经典考点。

---

### 1\. 核心结论 (Conclusion First)

在循环中拼接字符串，`StringBuilder` **是绝大多数场景下的最优选择**。

-   **单线程循环/局部变量：** 必须使用 `StringBuilder`。
    
-   **多线程共享变量：** 使用 `StringBuffer`（虽然这种情况极少见）。
    
-   **禁止在循环内使用** `+`**：** 这会导致严重的性能问题和内存压力。
    

---

### 2\. 深度解析 (Detailed Breakdown)

#### ① String：不可变的“沉重枷锁”

`String` 在 Java 中是不可变的（immutable）。当你写 `str = str + "a"` 时，JVM 并不是在原字符串后面加了字符，而是：

1.  创建一个临时 `StringBuilder`。
    
2.  调用 `append`。
    
3.  调用 `toString` 生成一个**全新**的 `String` 对象。 **结论：** 在循环中使用 `+`，每循环一次都会在堆上创建新的对象。如果循环 1 万次，就会产生 1 万个 `StringBuilder` 和 1 万个 `String` 临时对象，极大地触发 GC（垃圾回收），降低程序响应速度。
    

#### ② StringBuilder：轻量级的“高效画板”

`StringBuilder` 是专门为字符串拼接设计的可变对象。它内部维护了一个 `char[]` 数组（Java 9+ 变为 `byte[]`），拼接时直接操作数组。

-   **优点：** 不产生中间多余对象，速度极快。
    
-   **缺点：** 非线程安全（方法没有 `synchronized` 关键字）。
    

#### ③ StringBuffer：带锁的“安全卫士”

`StringBuffer` 的 API 与 `StringBuilder` 完全一致，唯一的区别是它的主要方法都加了 `synchronized` 锁。

-   **优点：** 线程安全。
    
-   **缺点：** 频繁加锁和释放锁会带来额外的性能损耗。
    
-   **现实情况：** 绝大多数拼接逻辑都发生在方法内部的局部变量中，不存在线程竞争，因此 `StringBuffer` 已经很少在现代开发中被作为首选。
    

---

### 3\. 面试官视角 (The "Interviewer's Lens")

**追问 1：编译器不是会自动把** `+` **优化成** `StringBuilder` **吗？为什么循环里还是不行？**

-   **陷阱：** 编译器的优化是“单行”的。
    
-   **真相：** 如果你在循环里写 `s += "a"`，编译器确实会帮你优化成 `StringBuilder`。但问题是，它会在**循环体内部**创建 `StringBuilder`。也就是说，循环 1000 次，它依然创建了 1000 个 `StringBuilder` 实例。而手动使用 `StringBuilder` 只需要在循环外创建 **1 个** 实例。
    

**追问 2：Java 8 之后和 Java 9 之后对字符串拼接有什么优化？**

-   Java 9 引入了 `StringConcatFactory`（使用 `invokedynamic` 动态生成拼接逻辑），提高了非循环场景下的拼接效率，但循环内的拼接依然建议显式使用 `StringBuilder`。
    

---

### 4\. 代码示例 (Code Snippet)

```java
// ❌ 错误示范：性能极差，容易导致 Full GC
String result = "";
for (int i = 0; i < 10000; i++) {
    result += i; // 每次循环都会 new StringBuilder() 和 toString()
}

// ✅ 正确示范：高效且专业
StringBuilder sb = new StringBuilder(); // 如果知道大概长度，可以指定容量：new StringBuilder(10000);
for (int i = 0; i < 10000; i++) {
    sb.append(i);
}
String finalResult = sb.toString();
```

---

### 5\. 核心总结 (Summary)

1.  **性能优先：** 循环拼接必用 `StringBuilder`，通过减少对象创建降低 GC 频率。
    
2.  **线程安全：** 仅在多个线程同时操作同一个字符串缓冲区时才考虑 `StringBuffer`。
    
3.  **预分配空间：** 如果能预估拼接后的长度，给 `StringBuilder` 构造函数传入 `capacity`，可以避免内部数组频繁扩容（类似 ArrayList 扩容）。
