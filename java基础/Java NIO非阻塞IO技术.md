Java 的 **NIO**（New Input/Output，在某些语境下也叫 Non-blocking IO，即**非阻塞 IO**）是 Java 1.4 引入的一套新的 IO 标准。

如果要用一句话总结：**NIO 是为了解决传统 BIO（Blocking IO，阻塞式 IO）在处理高并发连接时性能低下的问题。**

---

### 1\. 为什么要用 NIO？（BIO vs NIO）

在传统的 **BIO** 中，当你写一个网络服务器：

-   **模式**：一个线程处理一个连接（One Thread per Connection）。
    
-   **问题**：如果有一万个客户端连接，服务器就得开一万个线程。
    
-   **阻塞**：如果客户端连接了但不发数据，这个线程就会一直死等（阻塞），浪费 CPU 和内存资源。
    

在 **NIO** 中：

-   **模式**：一个线程可以处理多个连接（IO 多路复用）。
    
-   **特点**：线程不再死等数据，而是“有事儿再叫我”。
    

---

### 2\. NIO 的三大核心组件

理解 NIO，只需记住这三个词：**Buffer（缓冲区）**、**Channel（通道）**、**Selector（选择器）**。

#### ① Buffer (缓冲区)

-   **概念**：在 BIO 中，数据是直接从流里读写的；在 NIO 中，所有数据必须经过 Buffer。
    
-   **本质**：它就是一个数组，但提供了对数据的结构化访问，可以跟踪读写位置。
    
-   **常见类型**：`ByteBuffer`（最常用）, `IntBuffer`, `CharBuffer` 等。
    

#### ② Channel (通道)

-   **概念**：它是双向的“管道”。在 BIO 中，`InputStream` 只能读，`OutputStream` 只能写；而 NIO 的 `Channel` 既可以读也可以写。
    
-   **关系**：Channel 就像铁路，Buffer 就像火车。数据总是从 Channel 读到 Buffer，或者从 Buffer 写入 Channel。
    

#### ③ Selector (选择器) —— **NIO 的灵魂**

-   **概念**：Selector 是一个调度员。它允许**单个线程**同时监控多个 Channel 的 IO 状况（比如：哪个连接有新数据进来了？哪个连接断开了？）。
    
-   **机制**：Channel 会向 Selector 注册自己感兴趣的事件（如 `OP_READ`）。Selector 会不断轮询这些 Channel，当某个事件发生时，它才通知线程去处理。
    

---

### 3\. 形象的比喻：去餐厅吃饭

-   **BIO (阻塞式)**： 每个客人都配一个专属服务员。如果客人一直看菜单不点菜，服务员就站在旁边死等，不能去服务别人。这就是“一连接一线程”，效率极低。
    
-   **NIO (非阻塞式)**： 餐厅只有一个或几个服务员（线程）。
    

1.  客人（Channel）来了，先拿个菜单（注册事件）。
    
2.  服务员（Selector）在转圈巡视。
    
3.  如果客人举手说“我选好了”（触发事件），服务员才过去收菜单（处理数据）。
    
4.  收完菜单，服务员立即去巡视其他桌，不用在那等。
    

---

### 4\. NIO 的优缺点

#### 优点：

1.  **高并发能力强**：单线程就能管理成千上万个连接，极大减少了线程切换和内存开销。
    
2.  **非阻塞**：读写操作不会让线程卡死。
    

#### 缺点：

1.  **编程难度大**：直接使用 Java 原生 NIO API 非常痛苦，需要处理各种状态、断连重连、半包黏包等问题。
    
2.  **处理时间**：如果单个请求的处理逻辑非常重（比如耗时很久的计算），依然会卡住 Selector 线程。
    

---

### 5\. 平时在哪里能见到 NIO？

虽然你可能不会直接写 `Selector.open()`，但你用的框架几乎都在用它：

1.  **Netty**：目前 Java 界最流行的网络通信框架，底层就是对 NIO 的完美封装。
    
2.  **Tomcat (8.0+)**：现代版本的 Tomcat 默认使用 NIO 模式来处理 HTTP 请求。
    
3.  **消息队列**：Kafka、RocketMQ 的高性能通信底层。
    
4.  **分布式协调**：Zookeeper。
    

### 总结

-   **BIO** 适合：连接数少且固定的架构，编程简单。
    
-   **NIO** 适合：连接数多且连接时间短的架构（如聊天服务器、RPC 框架、高性能中间件）。
    

**面试常考点**：NIO 是**同步非阻塞**的。

-   **同步**：因为线程还是需要主动去询问 Selector 是否有数据。
    
-   **非阻塞**：因为如果没有数据，线程可以立刻去做别的事，不会被卡住。
