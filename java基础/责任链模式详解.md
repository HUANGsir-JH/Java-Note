**责任链模式（Chain of Responsibility Pattern）** 是一种行为型设计模式。

它的核心思想是：**将请求的发送者和接收者解耦，让多个对象都有机会处理这个请求。** 这些对象连成一条链，并沿着这条链传递请求，直到有一个对象处理它为止。

---

### 1\. 核心概念

在责任链模式中，每个处理器（Handler）都包含对“下一个处理器”的引用。当一个请求进入时：

1.  当前处理器判断自己是否能处理该请求。
    
2.  如果能处理，则处理并停止（或者继续向下传递，取决于具体需求）。
    
3.  如果不能处理，则转发给链中的下一个处理器。
    

### 2\. 生活中的例子：请假审批

假设你在公司请假：

-   **请假 1 天**：组长（Leader）审批即可。
    
-   **请假 3 天**：组长审批不了，转交给经理（Manager）。
    
-   **请假 7 天**：经理也审批不了，转交给总监（Director）。
    
-   **请假 15 天**：总监审批不了，转交给老板（CEO）。
    

你不需要关心最终是谁审批的，你只需要把请假单交给你的“组长”就行了，请求会自动在链条上传递。

---

### 3\. 代码实现（Java 示例）

#### 第一步：定义抽象处理者

```java
public abstract class Approver {
    protected Approver next; // 下一个审批者

    public void setNext(Approver next) {
        this.next = next;
    }

    // 处理请求的方法
    public abstract void processRequest(int leaveDays);
}
```

#### 第二步：实现具体的处理者

```java
// 组长
class Leader extends Approver {
    public void processRequest(int leaveDays) {
        if (leaveDays <= 2) {
            System.out.println("组长审批通过：" + leaveDays + "天");
        } else if (next != null) {
            next.processRequest(leaveDays);
        }
    }
}

// 经理
class Manager extends Approver {
    public void processRequest(int leaveDays) {
        if (leaveDays <= 7) {
            System.out.println("经理审批通过：" + leaveDays + "天");
        } else if (next != null) {
            next.processRequest(leaveDays);
        }
    }
}

// 老板
class CEO extends Approver {
    public void processRequest(int leaveDays) {
        System.out.println("老板审批通过：" + leaveDays + "天");
    }
}
```

#### 第三步：客户端组装链条并使用

```java
public class Client {
    public static void main(String[] args) {
        Approver leader = new Leader();
        Approver manager = new Manager();
        Approver ceo = new CEO();

        // 组装责任链
        leader.setNext(manager);
        manager.setNext(ceo);

        // 提交请求
        leader.processRequest(1);  // 组长处理
        leader.processRequest(5);  // 经理处理
        leader.processRequest(20); // 老板处理
    }
}
```

### 4\. 责任链模式的优缺点

#### 优点：

1.  **降低耦合度**：发送者不需要知道具体是谁处理了请求。
    
2.  **灵活性高**：可以动态地改变链内的成员或调整它们的次序（例如增加一个“副总监”级别）。
    
3.  **符合单一职责原则**：每个类只需关注自己的处理逻辑，不属于自己的就丢给别人。
    

#### 缺点：

1.  **性能问题**：如果链条非常长，请求可能会经过很多环节，影响性能。
    
2.  **不能保证被处理**：如果没有任何一个节点处理该请求，它可能会“掉到地上”（请求到链尾就结束了）。
    
3.  **调试困难**：由于请求在多个对象间传递，排查问题时可能需要跟踪多个类。
    

---

### 5\. 常见的实际应用场景

你在开发中其实经常遇到责任链模式：

1.  **Servlet 过滤器（Filter）**： 在 Java Web 中，`FilterChain` 就是典型的责任链。每个 Filter 可以决定处理请求、修改请求，或者调用 `chain.doFilter()` 传给下一个滤器。
    
2.  **Spring Security**： 权限校验是一层一层的（是否登录、是否有角色、是否有权限），每一层都是一个拦截器。
    
3.  **日志系统（Logback/Log4j）**： 日志级别（Debug -> Info -> Error）。如果设置了 Error 级别，Info 级别的处理器就会直接跳过，传给下一级。
    
4.  **异常处理（try-catch）**： 虽然不是标准的模式实现，但逻辑很像：如果当前 catch 块不能匹配异常，它会向上抛出，直到被处理或程序崩溃。
    

### 总结

**责任链模式**就像是在接力赛。每个人只管跑好自己那一棒，如果跑完了还没到终点，就把接力棒传给下一个人。它最适合用于**有多个步骤处理同一个请求，且处理步骤可能动态变化**的场景。
