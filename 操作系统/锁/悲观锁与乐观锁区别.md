
这是一个非常经典的面试题，它不仅考察你对**并发控制**的理解，更能体现你是否具备**根据业务场景选型**的工程能力。

简单来说：**悲观锁（Pessimistic Locking）**是“先锁后做”，**乐观锁（Optimistic Locking）**是“先做再看”。

---

### 1. Core Conclusion（一句话核心总结）
**悲观锁**假设冲突一定会发生，因此在操作数据前先加锁，确保自己独占资源（适合**写多读少**、强一致性场景）；**乐观锁**假设冲突很少发生，在更新时才检查数据是否被他人改动过（适合**读多写少**、高并发、锁竞争少的场景）。

---

### 2. Deep Dive（深度剖析：策略的权衡）

#### 🛡️ 悲观锁：防患于未然
* **心理预期**：总有刁民想害朕（总有别的线程想改我的数据）。
* **核心逻辑**：
 1. 申请锁。
 2. 拿到锁后，开始操作数据。
 3. 操作完，释放锁。
* **开销点**：数据库层面的行锁/表锁，或者编程语言层面的 `Mutex/synchronized`。由于涉及线程挂起和唤醒，**上下文切换开销大**。
* **适用场景**：
 * **写操作频繁**：如果冲突概率极高，乐观锁会频繁失败重试，反而浪费 CPU。
 * **强一致性要求**：比如银行转账，必须严格排队。

#### 🚀 乐观锁：信任加验证
* **心理预期**：大家都很守规矩，冲突很少见。
* **核心逻辑**：
 1. 不加锁，直接读取当前数据。
 2. 在内存中修改数据。
 3. **提交更新时**，检查版本号或时间戳：
 * 如果版本没变：说明期间没人动过，提交成功。
 * 如果版本变了：说明被别人捷足先登，提交失败，报错或触发**自旋重试**。
* **实现方式**：通常利用 **CAS (Compare And Swap)** 指令或数据库的 `version` 字段。
* **适用场景**：
 * **读多写少**：绝大多数请求只是看，极少数改。
 * **短事务/低竞争**：冲突概率极低，省去了加锁撤锁的开销，吞吐量极高。

### 3. The "Interviewer's Lens"（面试官视角：连环追问与高频踩坑）

**🔥 面试官连环追问：**
* **追问 1：乐观锁如果不加锁，它是怎么保证原子性的？**
 * *高分回答：* 在 CPU 层面，依靠 **CAS 指令**（如 x86 的 `LOCK CMPXCHG`）。它在硬件层面保证了“比较-更新”这两个动作是原子的，不可分割。
* **追问 2：谈谈乐观锁最著名的 ABA 问题，以及怎么解决？**
 * *高分回答：*
 * **现象**：线程 1 读取值为 A。线程 2 把 A 改成 B，又改回 A。线程 1 提交时发现还是 A，以为没变，更新成功。
 * **风险**：虽然值没变，但过程变了（比如链表头结点变了）。
 * **解法**：增加**版本号（Version）**或**时间戳**。不仅比较值，还比较版本（1A -> 2B -> 3A）。
* **追问 3：自旋锁（Spinlock）属于悲观锁还是乐观锁？**
 * *高分回答：* **自旋锁在实现上是“悲观”的**。因为它虽然不让出 CPU，但它的目的是为了**排他（独占资源）**。只要是锁（Lock），其语义都是悲观的。乐观锁通常被称为“无锁（Lock-free）”机制。

**💀 常见踩坑（Pitfalls）：**
* **误区：** “乐观锁性能一定比悲观锁好。”
 * *纠正：* 在**高竞争（High Contention）**环境下，乐观锁会因为不断地 CAS 失败而陷入死循环（自旋），空转 CPU，此时性能远不如直接把线程挂起的悲观锁。

### 4. Code / Practical Examples（实战演练：两种实现）

#### 💾 数据库场景 (SQL)
* **悲观锁（使用 `FOR UPDATE`）**:
 ```sql
 -- 事务 A
 SELECT amount FROM account WHERE id = 1 FOR UPDATE; -- 此时这行被锁死，别人查不了也改不了
 UPDATE account SET amount = amount - 100 WHERE id = 1;
 COMMIT;
 ```
* **乐观锁（使用版本号）**:
 ```sql
 -- 1. 先查出当前版本
 SELECT amount, version FROM account WHERE id = 1; -- 假设 version = 10
 -- 2. 更新时带上版本检查
 UPDATE account SET amount = amount - 100, version = version + 1
 WHERE id = 1 AND version = 10; -- 如果返回 0 行，说明已被他人修改，需重试
 ```

#### 💻 编程语言场景 (C++ / Java)
* **悲观锁**: `std::mutex` 或 `std::lock_guard`。
* **乐观锁**: `std::atomic<int>`。
 ```cpp
 std::atomic<int> count{0};
 void increment() {
 int old_val = count.load();
 // 尝试更新，如果 count 依然等于 old_val，就更新为 old_val + 1
 // 如果不等（被别人改了），compare_exchange_weak 返回 false，继续循环
 while (!count.compare_exchange_weak(old_val, old_val + 1));
 }
 ```

### 5. Summary（三点总结）
1. **悲观锁**：适合**重冲突、强一致性**场景。缺点是上下文切换开销大，可能导致阻塞。
2. **乐观锁**：适合**低竞争、高并发**场景。通过 CAS 或版本号实现，无锁开销，但存在 ABA 问题和高竞争下的 CPU 消耗。
3. **选型依据**：冲突概率。低冲突用乐观，高冲突用悲观。
