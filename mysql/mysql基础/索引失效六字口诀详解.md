
这是一个面试中的超高频问题。为了方便记忆，我总结了一个六字口诀：**“模、型、数、运、最、快”**。

### 1. 核心结论 (Core Conclusion)
索引失效的本质是：**查询条件破坏了 B+ 树索引的有序性，或者优化器评估全表扫描的成本更低。** 当数据库无法通过索引树进行二分查找定位数据时，索引就会退化为全表扫描。

---

### 2. 深度剖析 (Detailed Breakdown) —— 索引失效六大场景

#### ① 模：模糊查询 (Left-prefix Wildcard)
* **现象：** `LIKE '%abc'` 以百分号开头。
* **原理：** B+ 树是按字符串从左到右排序的。开头都不确定，索引树根本无法比较大小，只能挨个扫。
* **注意：** `LIKE 'abc%'`（后缀模糊）是可以走索引的。

#### ② 型：类型转换 (Type Mismatch)
* **现象：** 字段是字符串 `varchar`，查询写成 `WHERE code = 123`（没加引号）。
* **原理：** MySQL 会隐式调用函数将字符串转为数字。一旦在索引列上发生函数转换，索引失效。
* **反直觉：** 如果字段是数字，查 `WHERE id = '123'`，索引**通常不会**失效，因为 MySQL 把常量字符串转成了数字。

#### ③ 数：函数运算 (Functions)
* **现象：** `WHERE ABS(amount) = 100` 或 `WHERE DATE(create_time) = '2023-01-01'`。
* **原理：** 索引树里存的是原始值，不是计算后的值。计算后的结果在树里是无序的。

#### ④ 运：表达式运算 (Operators)
* **现象：** `WHERE id + 1 = 10`。
* **原理：** 同样道理，索引列参与了算术运算。
* **修正：** 改写为 `WHERE id = 10 - 1`，索引恢复。

#### ⑤ 最：最左前缀原则 (Leftmost Prefix)
* **现象：** 联合索引 `(a, b, c)`，查询 `WHERE b = 1 AND c = 2`。
* **原理：** 跳过了第一列 `a`，后面的 `b` 和 `c` 在全局范围内是无序的。

#### ⑥ 快：范围查询及优化器成本 (Cost Estimation)
* **现象：** `WHERE id > 0`（假设 ID 是正数）。
* **原理：** 如果优化器发现符合条件的记录超过了全表的 **20%~30%**，它认为回表的随机 IO 开销比全表扫描还大，于是放弃索引。这叫“优化器由于扫描行数过多导致的索引失效”。

---

### 3. 面试官视角 (The Interviewer's Lens)

这部分是考察你是否真的有**实战经验**：

* **陷阱 1：`OR` 条件。**
 * “如果 `a` 有索引，`b` 没索引，执行 `WHERE a=1 OR b=2` 索引生效吗？”
 * *答案：* **失效**。因为 `b` 没索引，必须全表扫描，那 `a` 走索引也没意义了。除非 `a` 和 `b` 都有索引，可能会触发 `index_merge`（索引合并）。
* **陷阱 2：`IS NULL` 和 `IS NOT NULL`。**
 * “很多人说 `IS NULL` 不走索引，对吗？”
 * *答案：* **不对**。在 InnoDB 中，索引是可以存储 NULL 值的。是否走索引取决于**数据分布**（优化器评估）。如果表中 90% 的数据都是 NULL，查 `IS NULL` 就不走索引；反之则走。
* **陷阱 3：不等于 `!=` 或 `<>`。**
 * “不等于一定会导致索引失效吗？”
 * *答案：* **不一定**。同样取决于优化器。如果“不等于某值”的记录只占全表极小部分，索引依然可能生效。

---

### 4. 代码示例 (Code Snippet)

```sql
-- 联合索引 (name, age)

-- ✅ 有效：最左匹配
SELECT * FROM users WHERE name = '张三';

-- ❌ 失效：左模糊
SELECT * FROM users WHERE name LIKE '%三';

-- ❌ 失效：对索引列使用函数
SELECT * FROM users WHERE UPPER(name) = 'ZHANGSAN';

-- ❌ 失效：类型转换（假设 code 是 varchar）
SELECT * FROM users WHERE code = 1001; 

-- ✅ 有效：将运算移到等号右边
SELECT * FROM users WHERE age = 20 - 1;
```

---

### 5. 总结 (Summary)

1. **记住口诀：** 模、型、数、运、最、快。
2. **核心准则：** 不要在索引列上做任何“手脚”（运算、函数、转换）。
3. **终极武器：** 不确定的情况下，直接用 `EXPLAIN` 看执行计划。
