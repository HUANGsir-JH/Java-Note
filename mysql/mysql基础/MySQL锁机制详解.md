
MySQL 的锁机制是数据库并发控制的核心。为了在“高并发”和“数据一致性”之间取得平衡，MySQL 设计了一套非常复杂的锁体系。

---

### 1. 核心结论 (Core Conclusion)
MySQL 的锁主要由 **存储引擎** 实现。**MyISAM 只支持表锁**，而 **InnoDB 支持行锁和表锁**。InnoDB 的强大之处在于它通过 **MVCC（多版本并发控制）** 实现了“读不加锁”，并通过 **Next-Key Lock（临键锁）** 在可重复读隔离级别下解决了幻读问题。

---

### 2. 深度剖析：锁的分类与级别

我们可以从三个维度来拆解 MySQL 的锁：

#### ① 按粒度分类（Granularity）
* **表级锁 (Table Lock)**：开销小，加锁快；不会出现死锁；锁定粒度大，发生锁冲突的概率最高，并发度最低。常用于 DDL 语句。
* **行级锁 (Row Lock)**：开销大，加锁慢；**会出现死锁**；锁定粒度最小，发生锁冲突的概率最低，并发度也最高。
* **页级锁**：介于表锁和行锁之间（BDB 引擎支持，现在很少见）。

#### ② 按兼容性分类（Compatibility）
* **共享锁 (S Lock / 读锁)**：允许一个事务去读一行，阻止其他事务获得相同数据集的排他锁。
* **排他锁 (X Lock / 写锁)**：允许获取排他锁的事务更新数据，阻止其他事务获取相同数据集的共享读锁和排他写锁。

#### ③ InnoDB 特有的行锁模式（面试重难点）
InnoDB 的行锁是通过给 **索引上的索引项** 加锁来实现的，而不是给记录加锁。
1. **记录锁 (Record Lock)**：单个行记录上的锁。
2. **间隙锁 (Gap Lock)**：锁定一个范围，但不包括记录本身（防止幻读）。
3. **临键锁 (Next-Key Lock)**：Record Lock + Gap Lock，锁定一个范围并锁定记录本身。**这是 InnoDB 默认的锁算法**。

#### ④ 意向锁 (Intent Locks - 表级)
* **意向共享锁 (IS)** / **意向排他锁 (IX)**
* **为什么要它？** 它是为了快速判断表里是否有行被锁住。如果你想给表加“表锁”，不需要去一行行查有没有行锁，直接看表上的“意向锁”标志位即可。

---

### 3. 代码演示：如何手动加锁

```sql
-- 开启事务
BEGIN;

-- 加共享锁（读锁），其他事务可读不可改
SELECT * FROM users WHERE id = 1 LOCK IN SHARE MODE;

-- 加排他锁（写锁），其他事务不可读（指当前读）不可改
SELECT * FROM users WHERE id = 1 FOR UPDATE;

-- 提交事务，释放锁
COMMIT;
```

### 4. 面试官视角：深度连环炮 (Interviewer's Lens)

**问 1：InnoDB 既然支持行锁，为什么有时候会锁住整张表？**
* **坑点**：InnoDB 的行锁是加在**索引**上的。
* **答案**：如果你的 `UPDATE` 或 `DELETE` 语句的 `WHERE` 条件**没有走索引**，那么 InnoDB 只能进行全表扫描。为了保证并发安全，它会给全表的所有记录都加上锁（实际上是把所有聚簇索引项都锁了），表现得就像表锁一样。

**问 2：什么是死锁？如何解决？**
* **原因**：两个或多个事务在执行过程中，因争夺资源而造成的一种互相等待的现象（环路等待）。
* **解决方法**：
 1. **超时机制**：`innodb_lock_wait_timeout`。
 2. **死锁检测**：InnoDB 有 `wait-for graph` 算法，发现死锁后会主动回滚持有锁最少的事务。
 3. **业务规避**：固定的顺序访问表和行；避免大事务。

**问 3：乐观锁和悲观锁在 MySQL 中怎么体现？**
* **悲观锁**：就是上面说的 `FOR UPDATE`，假定一定会冲突，先抓到锁再说。
* **乐观锁**：MySQL 本身没有乐观锁。通常由业务实现，通过 `version` 字段或时间戳，在 `UPDATE ... WHERE version = old_version` 时判断。

---

### 5. 总结 (Summary)

1. **MyISAM 选表锁，InnoDB 选行锁**。
2. **行锁依赖索引**：没索引，行锁变“表锁”。
3. **读写分离**：大部分 `SELECT` 是快照读（不加锁），只有 `FOR UPDATE` 是当前读（加锁）。
4. **间隙锁**：是 InnoDB 解决幻读的杀手锏。
