双亲委派机制（Parents Delegation Model）是 Java 类加载机制的**精髓**。如果把类加载器比作一个职场体系，双亲委派就是一套“**遇事上报，逐级审批**”的办事流程。

---

### 1\. 核心结论 (Core Conclusion)

**双亲委派机制的工作原理概括为：** 当一个类加载器收到类加载请求时，它首先不会尝试自己去加载这个类，而是把请求**委派给父类加载器**去完成。每一层都是如此，因此所有的加载请求最终都应该传送到顶层的\*\*启动类加载器（Bootstrap ClassLoader）\*\*中。只有当父加载器反馈自己无法完成加载（在它的搜索范围中没找到所需的类）时，子加载器才会尝试自己去加载。

---

### 2\. 深度剖析 (Deep Breakdown)

#### A. 工作流程（三部曲）

1.  **检查：** 先看自己是否已经加载过这个类（缓存检查）。
    
2.  **委派：** 如果没加载过，调用父加载器的 `loadClass()` 方法。
    
3.  **加载：** 如果父加载器也找不到，才由当前加载器调用自己的 `findClass()` 方法。
    

#### B. 形象比喻：向领导请假

-   **你（Custom ClassLoader）：** 收到一份文件。你不敢直接签字，先交给经理。
    
-   **经理（AppClassLoader）：** 也不敢签，交给总监。
    
-   **总监（ExtClassLoader）：** 也不敢签，交给老板。
    
-   **老板（BootstrapClassLoader）：** 看了一下，如果是《公司章程》（核心库），他就签了。如果不是，他就退回给总监。
    
-   **逐级退回：** 如果大家都不认识，最后只能你自己签。如果你也签不了（找不到文件），就报 `ClassNotFoundException`。
    

#### C. 为什么要这样做？（核心价值）

1.  **安全性（Security）：** 防止核心 API 被篡改。如果没有此机制，用户自己写一个 `java.lang.String` 放在 ClassPath 下，JVM 可能会加载这个伪造的类。有了委派机制，`String` 永远由最顶层的老板（Bootstrap）加载，确保加载的是官方核心库。
    
2.  **唯一性（Uniqueness）：** 保证同一个类在同一个类加载器环境中只被加载一次。
    

---

### 3\. 面试官视角 (Interviewer's Lens)

**经典追问一：如何“破坏”双亲委派？**

-   **方法：** 自定义类加载器，并重写 `loadClass` 方法（默认它是委派逻辑的所在地）。如果你重写的是 `findClass`，则依然遵循委派模式。
    
-   **典型场景：**
    
-   **JDBC (SPI 机制)：** 核心包在顶层，但实现包（Driver）在底层。老板需要请下属办事，通过 `Thread Context ClassLoader` 实现逆向调用。
    
-   **Tomcat：** 为了实现 Web 应用间的隔离（两个应用可以用不同版本的同一个类库），Tomcat 打破了委派机制，优先让自己的加载器去加载。
    

**经典追问二：源码里是怎么体现的？**

-   *答案：* 就在 `java.lang.ClassLoader.loadClass()` 方法里。它的逻辑非常清晰：先调 `findLoadedClass()` -> 调 `parent.loadClass()` -> 最后调自己的 `findClass()`。
    

---

### 4\. 源码精简版 (Understanding the Code)

这是 `ClassLoader.loadClass` 的核心逻辑伪代码：

```java
protected Class<?> loadClass(String name, boolean resolve) {
    synchronized (getClassLoadingLock(name)) {
        // 1. 先检查类是否已经被加载过了
        Class<?> c = findLoadedClass(name);
        if (c == null) {
            try {
                if (parent != null) {
                    // 2. 如果有父类，委派给父类加载
                    c = parent.loadClass(name, false);
                } else {
                    // 3. 如果父类为空，说明到了顶层，委派给启动类加载器
                    c = findBootstrapClassOrNull(name);
                }
            } catch (ClassNotFoundException e) {
                // 父类加载器抛出异常，说明它无法加载
            }

            if (c == null) {
                // 4. 万般无奈之下，才调用自己的 findClass 去加载
                c = findClass(name);
            }
        }
        return c;
    }
}
```

### 5\. 总结 (Summary)

-   **向上委派：** 是一种“责任上缴”的防守策略。
    
-   **安全防线：** 确保了 Java 核心生态的稳定，是 Java “长治久安”的基础。
    
-   **并非绝对：** 它是推荐模型，不是强制约束。在需要灵活性的场景（如热部署、模块隔离）下，可以主动打破。
