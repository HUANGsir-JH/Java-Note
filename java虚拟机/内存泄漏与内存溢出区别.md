这是一个非常基础但又极其核心的问题。很多开发者会将两者混淆，但实际上它们的**成因**和**性质**完全不同。

---

### 1\. 核心结论 (Core Conclusion)

-   **内存泄漏 (Memory Leak)：** 是指程序申请了内存，但**无法释放**已不再使用的对象。它就像水桶有个小洞，水一点点流失，最终导致水桶变空（可用内存枯竭）。
    
-   **内存溢出 (Out Of Memory - OOM)：** 是指程序申请内存时，**没有足够的内存**供其使用。它就像往水桶里倒水，水已经满了还要继续倒，最终水溢出来了。
    
-   **关系：** **内存泄漏的持续堆积，最终会导致内存溢出。**
    

---

### 2\. 深度剖析 (Deep Breakdown)

#### A. 内存泄漏 (Memory Leak)

-   **本质：** 对象在逻辑上已经没用了，但在语法上仍然被**强引用**关联着，导致垃圾回收器（GC）无法回收它。
    
-   **常见场景：**
    

1.  **静态集合类：** `static Map` 存放了大量对象且从未 `remove`，这些对象会随类一直存在。
    
2.  **未关闭的资源：** 数据库连接、网络 IO、File 流不调用 `close()`。
    
3.  **ThreadLocal：** 线程复用且未调用 `remove()`（正如我们刚才聊到的）。
    
4.  **内部类引用：** 非静态内部类会隐式持有外部类的引用，导致外部类无法回收。
    

-   **特征：** 内存占用呈**阶梯状或斜坡状**上升，重启能暂时解决，但运行一段时间后必定复发。
    

#### B. 内存溢出 (Memory Overflow / OOM)

-   **本质：** 内存不够用了。JVM 尝试 GC 后依然无法获取足够的空间。
    
-   **常见类型：**
    

1.  `java.lang.OutOfMemoryError: Java heap space`：堆溢出。创建了太多大对象或内存泄漏堆积。
    
2.  `java.lang.StackOverflowError`：栈溢出。通常是递归太深或死循环调用。
    
3.  `java.lang.OutOfMemoryError: Metaspace`：元空间溢出。加载了太多的类（如动态代理生成过多）。
    

-   **特征：** 程序直接崩溃，抛出 Error，无法继续运行。
    

---

### 3\. 面试官视角 (Interviewer's Lens)

**经典追问一：如果线上发生了 OOM，你的排查思路是什么？**

1.  **先看日志：** 确认是哪种 OOM（堆、栈还是元空间）。
    
2.  **获取 Dump 文件：** 通过参数 `-XX:+HeapDumpOnOutOfMemoryError` 自动生成，或手动执行 `jmap -dump`。
    
3.  **使用分析工具：** 使用 **MAT (Memory Analyzer Tool)** 或 **JProfiler** 打开 Dump 文件。
    
4.  **定位元凶：** 查找 **Biggest Objects**，看是哪个类的实例占用了巨额内存，追踪其引用链（GCRoots），判断是正常业务压力还是内存泄漏。
    

**经典追问二：内存泄漏和内存溢出，哪个更难排查？**

-   *答案：* **内存泄漏更难。** OOM 发生时通常伴随 Crash 和报错信息。而内存泄漏在早期是无感的，只会导致系统越来越卡，GC 越来越频繁，直到最后爆发 OOM。

### 4\. 代码案例：制造一个事故

**内存泄漏案例（静态集合）：**

```java
public class LeakExample {
    private static final List<Object> CACHE = new ArrayList<>();

    public void process(Object data) {
        CACHE.add(data); // 只是不断添加，从不清理
        // 逻辑处理完毕，但 data 对象因为被 CACHE 引用，永远无法回收
    }
}
```

**内存溢出案例（瞬间暴毙）：**

```java
public void oomExample() {
    List<byte[]> list = new ArrayList<>();
    while (true) {
        // 瞬间申请大量内存，直到把堆撑爆
        list.add(new byte[1024 * 1024 * 100]); // 每次 100MB
    }
}
```

---

### 5\. 总结 (Summary)

| 维度  | 内存泄漏 (Leak) | 内存溢出 (OOM) |
| --- | --- | --- |
| **形象比喻** | 占着茅坑不拉屎 | 茅坑不够用了 |
| **主要成因** | 代码逻辑错误（该断的引用没断） | 对象过多、参数设置太小、内存泄漏 |
| **解决难度** | 较高（需分析引用链） | 较低（通常通过调参数或增加资源解决） |
| **对系统的影响** | 运行速度逐渐变慢 | 系统直接挂掉 |
