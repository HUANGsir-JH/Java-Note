这是一个非常经典的 Agent 工程实战问题。在面试中，初级开发者通常会回答“修改 Prompt 警告大模型不要乱输出”，但资深架构师深知：**LLM 是概率模型，单纯靠 Prompt 永远无法实现 100% 的格式稳定。**

处理 Tool-use（工具调用）格式不合规，必须从系统架构层面设计\*\*“防御性编程与自我修复闭环”\*\*。

---

### 1\. 核心结论 (Conclusion First)

解决 Agent 工具调用格式不合规的问题，必须采用**三道防线架构**：

1.  **事前预防（Constrained Decoding / 结构化输出）**：在生成阶段强制锁死输出格式。
    
2.  **事中容错（Heuristic Parsing）**：使用正则和启发式规则，从脏数据中“抢救”出 JSON。
    
3.  **事后恢复（Self-Correction Loop）**：将程序的报错信息作为反馈（Feedback）扔回给大模型，让其自主修正并重试。
    

---

### 2\. 详细拆解 (Deep Dive)

#### 防线一：事前预防 (Prevention) —— 让它“无法”生成错误格式

这是目前最高效的解法。与其事后检查，不如在模型吐字（Decoding）时就限制它的词表。

-   **对于闭源 API (如 OpenAI/Anthropic)：** 必须开启最新的 **Structured Outputs (严格结构化输出)** 功能（如 OpenAI 的 `strict: true`）。底层原理是在解码阶段将 JSON Schema 转换为上下文无关文法（CFG），动态屏蔽（Token Masking）不合法的 Token 概率。
    
-   **对于开源/私有部署模型 (vLLM/Ollama)：** 必须在推理引擎层接入 **Outlines** 或 **Xorbits Inference** 等约束解码库。强制模型只能按照预设的 Pydantic 模型生成。
    

#### 防线二：事中容错 (Fault-Tolerant Parsing) —— “抢救”脏数据

很多时候，模型其实生成了正确的参数，但它“话太多”（比如在 JSON 前面加上了“好的，我将调用工具：”）。

-   **正则表达式提取：** 绝不能直接对 LLM 的原始文本调用 `json.loads()`。必须先用正则提取 `json ...` 代码块内的内容，或者匹配第一对 `{` 和最后一对 `}` 之间的字符串。
    
-   **自动补全缺失符号：** 使用第三方库（如 `json_repair`），自动修复缺失的引号、漏掉的逗号或未闭合的括号。
    
-   *类比：* 就像浏览器解析不规范的 HTML 网页一样，尽量做到“宽容输入”。
    

#### 防线三：事后恢复 (Self-Correction Loop) —— Agent 核心能力

当容错解析也失败，或者 Pydantic 校验发现参数类型错误（如要把字符串传给整型字段）、甚至产生**参数幻觉**（捏造了不存在的参数）时，触发反思重试。

-   **捕获异常并反馈：** 将 Python 抛出的 `ValidationError` 甚至完整的 Error Traceback 包装成一条 `ToolMessage` 发回给大模型。
    
-   **角色扮演修正：** “就像编译器给程序员报 Syntax Error 一样，明确告诉 LLM 错在哪一行、缺了什么字段，让它自己改 Bug”。

### 3\. 面试官视角 (Interviewer's Lens)

如果你提出了“自我修复闭环”，面试官必定会紧追不舍，考察你对边缘情况（Edge Cases）的把控。

#### 致命追问 1：如果大模型陷入了“死循环”（反复生成相同的错误格式，反复触发重试），你的系统怎么处理？

-   **架构师回答：** 任何重试机制都必须有 **熔断器 (Circuit Breaker)**。我们在 Agent 引擎层设置一个绝对的 `max_retries`（通常设为 3）。一旦触发阈值，不仅要抛出 `ToolExecutionException` 中断当前节点，还应该设计一个 **降级策略 (Fallback)**——比如把状态移交给“人工干预节点 (Human-in-the-loop)”，或者返回一个友好的错误提示让用户换种说法，绝不让引擎卡死。
    

#### 致命追问 2：大模型生成的格式是对的（JSON 合法），但是它传了一个业务逻辑上完全错误的参数（比如查询天气，传入了 City="火星"），导致外部 API 报错。这时候怎么办？

-   **架构师回答：** 这超出了 JSON 解析的范畴，属于**业务约束幻觉**。解法同样是利用闭环：外部 API 必须返回清晰的错误信息（如 `{"error": "城市不存在，可用列表为[北京, 上海...]"}`）。Agent 引擎捕获这个 API 业务报错，将其作为一次普通的工具调用结果返回给 LLM，LLM 的内在推理能力会理解这个业务反馈，并在下一步尝试换一个城市名称。

### 4\. 代码演示 (Python: 健壮的重试闭环机制)

这段代码展示了如何结合 **Pydantic 类型校验**与 **LLM 反馈重试**，构建一个坚不可摧的 Tool 解析器：

```python
import json
import re
from pydantic import BaseModel, ValidationError

# 1. 定义严格的工具参数 Schema
class WeatherToolArgs(BaseModel):
    city: str
    unit: str # 必须是 "celsius" 或 "fahrenheit"

def extract_json_from_text(text: str) -> str:
    """容错层：用正则抢救 JSON"""
    match = re.search(r'\{.*\}', text, re.DOTALL)
    if match:
        return match.group(0)
    return text

def execute_tool_with_retry(llm_client, initial_prompt, max_retries=3):
    current_prompt = initial_prompt
    
    for attempt in range(max_retries):
        print(f"\n[Attempt {attempt+1}] 正在调用 LLM...")
        
        # 模拟 LLM 生成（实际中会调用大模型 API）
        # 假设第一次生成了带有废话和错误参数的输出
        # raw_response = llm_client.generate(current_prompt) 
        
        if attempt == 0:
            raw_response = "好的！我将调用工具。\n```json\n{\"city\": \"Beijing\", \"unit\": \"kelvin\"}\n```"
        else:
            raw_response = "{\"city\": \"Beijing\", \"unit\": \"celsius\"}" # 第二次学乖了
            
        print(f"LLM 原始输出: {raw_response}")
        
        try:
            # 1. 清洗与提取 (容错层)
            cleaned_json_str = extract_json_from_text(raw_response)
            
            # 2. 解析 JSON
            parsed_dict = json.loads(cleaned_json_str)
            
            # 3. Pydantic 严格业务校验
            # 第一次循环会在这里报错，因为 unit 是 'kelvin'，不在预期内（需通过 Pydantic validator，此处简化演示）
            if parsed_dict.get("unit") not in ["celsius", "fahrenheit"]:
                raise ValidationError("Field 'unit' must be 'celsius' or 'fahrenheit'")
                
            valid_args = WeatherToolArgs(**parsed_dict)
            print(f"✅ 工具参数解析成功: {valid_args}")
            
            # 真正执行工具逻辑...
            return valid_args
            
        except (json.JSONDecodeError, ValidationError, Exception) as e:
            error_msg = str(e)
            print(f"❌ 解析失败: {error_msg}")
            
            # 核心机制：将精确的错误堆栈反馈给大模型
            feedback = f"\n\n[系统反馈]: 你的上一次工具调用格式报错了。错误详情: {error_msg}。请修正你的 JSON 并重新输出。"
            current_prompt += raw_response + feedback
            
    print("🚨 触发最大重试次数，系统熔断！")
    return None

# 运行测试
execute_tool_with_retry(None, "请查询北京天气，调用工具返回 JSON。")
```

---

### 5\. 快速总结 (Summary)

1.  **优先使用约束解码：** 能在模型生成层解决的（Structured Outputs / Outlines），就不要拖到代码层解决。
    
2.  **永远不要信任原始输出：** 在执行 `json.loads` 之前，必须有一层基于正则或 `json_repair` 的容错清洗。
    
3.  **构建反馈反思循环：** 把解析错误（Parse Error）和业务错误（API Error）一视同仁，作为环境反馈输入给 Agent 进行自我修正，并设置最大重试熔断。
