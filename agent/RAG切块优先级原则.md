
在 RAG（检索增强生成）系统中，切块（Chunking）的优先级顺序通常遵循**“从宏观结构到微观语义，最后到物理硬切”**的原则。

为了获得最佳的检索效果，建议按照以下优先级顺序进行处理：

---

### 1. 第一优先级：基于文档结构的切块 (Structural Splitting)
**核心逻辑：** 尊重作者原本的逻辑组织。
* **Markdown/HTML：** 优先按照标题层级（`#`, `##`, `###`）切块。
* **代码：** 按照函数（Function）、类（Class）定义来切块。
* **法律/正式文档：** 按照条款（Article）、章节（Chapter）切块。
* **优势：** 这种方式能最大限度保留上下文的完整性，防止把一个问题的答案拆到两个块里。

### 2. 第二优先级：基于语义单位的切块 (Semantic Splitting)
**核心逻辑：** 确保每一块都在讲同一个“意思”。
* **段落切分：** 如果没有明确的结构标签，优先按段落（`\n\n` 或 `\n`）切分。
* **语义聚类：** 使用 Embedding 模型计算句子间的相似度。当相邻两句的相似度低于阈值时，认为话题发生了转折，在此处进行切断。
* **优势：** 检索到的内容主题集中，减少无关噪声。

### 3. 第三优先级：递归字符切块 (Recursive Character Splitting) - **最常用**
**核心逻辑：** 在满足长度限制的前提下，尽量寻找自然的断句点。
这是 LangChain 等框架默认的推荐方式（`RecursiveCharacterTextSplitter`）。其优先级排序通常是：
1. **双换行符 (`\n\n`)**：段落。
2. **单换行符 (`\n`)**：行。
3. **句号/感叹号/问号**：句子。
4. **空格**：单词。
5. **空字符串**：字符（最后的保底手段）。
* **优势：** 极其灵活，能自动适配各种篇幅，保证切块大小均匀且不跨越单词。

### 4. 第四优先级：固定长度切块 (Fixed-size Splitting)
**核心逻辑：** 强行按照 Token 或字符数切分。
* **做法：** 每 512 个 Token 切一块。
* **缺点：** 极易在句子中间切断，导致语义丢失。
* **建议：** 仅作为上述方法失效时的最后保底手段，或者在处理极度非结构化数据时使用。

---

### 关键建议与参数配置

在执行上述优先级时，必须配合以下两个关键参数：

1. **Chunk Overlap (重叠度)：**
 * **建议：** 设置 10%-25% 的重叠（例如 Chunk Size 500，Overlap 50-100）。
 * **目的：** 保证相邻块之间有语境衔接，防止关键信息刚好落在切分点上。

2. **Chunk Size (块大小)：**
 * **短块 (128-256 Tokens)：** 适合事实查询（如“某人的生日是多少”）。检索精度高，但上下文少。
 * **长块 (512-1024 Tokens)：** 适合推理、总结（如“分析某公司的财务状况”）。上下文丰富，但噪声多。

### 总结建议的实施流程：
1. **判断类型：** 是代码还是文档？（是 -> 用结构化切分）
2. **首选递归：** 使用 `RecursiveCharacterTextSplitter`。
3. **设定阈值：** 根据模型 Context Window 设定 Size（通常 512 是甜点位）。
4. **加入重叠：** 必须设置 Overlap。
5. **语义微调：** 如果预算允许，在递归切分的基础上引入语义相似度判断。
