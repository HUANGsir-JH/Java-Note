# 📚 RAG 文本分块（Chunking）策略与工程实践详细学习笔记

## 第一部分：核心观点剖析 —— 为什么分块决定 RAG 的“生死”？

在 RAG 管道中，分块（Chunking）是 ETL（提取、转换、加载）阶段的核心组件。行业公认**“分块决定了 RAG 质量的 70%”**，其底层逻辑在于：

1. **模型上下文窗口与成本的限制**：大语言模型（LLM）无法一次性稳定处理超长文本。合理的分块能将长文档切分为中等粒度片段，避免长输入推高 API 成本并稀释信息密度 [^4]。
2. **决定检索的“信噪比”**：分块是向量数据库真正搜索的对象。块太大，无关内容会稀释关键特征，导致相似度判别力下降（噪声大）；块太小，则容易“只命中词不命中义”，缺乏支撑大模型推理的语境 [^4]。
3. **语义连续性的生命线**：糟糕的分块（如一刀切断了“前提条件”和“最终结论”）会导致召回的片段呈现顺序错乱、信息残缺。大模型在面对支离破碎的提示词时，为了补全逻辑，极易产生“幻觉”（胡编乱造） [^3][^4]。

## 第二部分：八种主流分块策略深度解析（含进阶扩充）

在您总结的七种策略基础上，结合最新行业实践（如 Agentic Chunking），扩展为以下详细对比：

### 1. 固定大小分块 (Fixed-size Chunking)
* **核心原理**：按预设的 Token 数或字符数（如 512 tokens）强行切分文本，辅以一定比例的重叠（Overlap） [^2]。
* **参数建议**：`chunk_size` 设为 300-800 字；`overlap` 设为 10%-15% [^2][^4]。
* **优点**：实现极其简单，计算速度极快，无需调用任何 NLP 模型。
* **缺点**：极其容易破坏语义边界，经常把一句话或一个段落从中间劈开。
* **适用场景**：毫无结构的纯文本、日志文件、或者作为快速验证的基线（Baseline）方案 [^4]。

### 2. 基于句子分块 (Sentence-based Chunking)
* **核心原理**：使用 NLP 工具（如 NLTK、spaCy 或正则匹配 `。！？；`）先将文本切分为完整的句子，再将若干句子聚合成满足长度的块 [^2][^4]。
* **优点**：保证了最基本的语义单元完整，避免了“半句话”问题，对构建精准的 QA 问答对非常友好 [^2][^4]。
* **缺点**：句子长度差异大，可能导致块大小不均；中文分句处理相对复杂 [^4]。
* **适用场景**：法律法规、新闻公告、FAQ 等以单句为核心信息载体的文本 [^4]。

### 3. 基于段落/文档结构分块 (Structure-aware Chunking)
* **核心原理**：利用文档固有的 Markdown、HTML 标签（如 `<h1>` 到 `<h6>`，`<table>`，`<li>`）或 PDF 元数据作为天然分界线 [^2][^4]。
* **优点**：完美贴合人类行文逻辑。能确保表格与表头不分离、代码块与注释不脱节 [^4]。
* **缺点**：高度依赖前置的文档解析（Document Parsing）质量，对非结构化文档无效 [^2]。
* **适用场景**：排版规范的学术论文、技术手册、法律合同、财报 [^2][^4]。

### 4. 递归分割 (Recursive Chunking) —— **工业界最常用**
* **核心原理**：提供一组由粗到细的分隔符（如：段落 `\n\n` -> 换行 `\n` -> 句号 -> 空格）。优先按大粒度拆分，如果拆出来的块超过了 `chunk_size`，再用下一级更细的分隔符递归拆分 [^4]。
* **优点**：在“保持语义边界”和“控制块大小”之间取得了极佳的平衡，适应性极强，是 LangChain 等框架的默认推荐 [^2][^4]。
* **缺点**：对极度格式化的文本（如密集的数据表）处理效果一般 [^4]。

### 5. 语义分块 (Semantic Chunking)
* **核心原理**：先将文档拆为句子并计算 Embedding 向量。通过计算相邻句子间的余弦相似度，当相似度发生“断崖式下跌”（低于某阈值）时，判定话题发生转移，在此处进行切断 [^2][^4][^7]。
* **优点**：精度最高，能创建逻辑上高度内聚的切块，大幅减少无关噪声 [^2][^5]。
* **缺点**：需要在切分阶段就调用 Embedding 模型，计算开销大、速度慢 [^2]。
* **适用场景**：主题跳跃较多的长篇研报、白皮书 [^2][^4]。

### 6. 层次化分块 / 父子分段 (Hierarchical / Parent-Child Chunking) —— **高阶玩法**
* **核心原理**：多层级架构。将文档切成大块（父块，如 1000 tokens）提供上下文，再在父块内切出小块（子块，如 150 tokens）用于向量检索 [^6]。
* **工作流**：检索时匹配精准的“子块”，但最终喂给大模型的是该子块所属的完整“父块” [^4][^6]。
* **优点**：既实现了极高的检索召回率（小块匹配准），又有效压制了幻觉（大块上下文全） [^6]。
* **缺点**：系统实现复杂度高，需要维护父子映射关系 [^4][^6]。

### 7. 智能体分块 (Agentic Chunking) —— **前沿探索**
* **核心原理**：使用一个小参数 LLM（Agent）模拟人类阅读，根据语义、逻辑和特定任务目标（如“为了生成摘要”或“为了代码问答”）动态决定在哪里切断 [^4][^5]。
* **优点**：灵活性和针对性极高，能处理结构极其混乱的复杂文档 [^2][^4]。
* **缺点**：Token 成本极高，处理速度极慢，目前多用于离线精细化知识库构建 [^2][^4]。

## 第三部分：实战建议与避坑指南 (Engineering Best Practices)

在将 RAG 从 Demo 推向生产环境的过程中，单纯选对策略还不够，必须注意以下工程细节：

### 1. 核心参数避坑：重叠度 (Overlap) 绝不是越大越好
* **必须设置重叠**：相邻块之间必须有重叠，以防止关键信息（如“张三的生日是 / 1990年”）被切断 [^2]。
* **黄金比例**：重叠量应严格控制在分块长度的 **10% 到 20%** 之间 [^2][^4]。
* **避坑**：超过 30% 的重叠会导致向量数据库体积暴增，检索时召回大量重复废话，不仅浪费大模型的 Context 窗口，还会干扰模型的推理注意力 [^4]。

### 2. 注入元数据 (Metadata Enrichment)：拯救孤立分块的利器
* 当一个块被单独检索出来时，它往往丧失了全局语境（例如，块里写着“本政策自即日起废止”，但没写是哪个政策）。
* **最佳实践**：在分块时，将文档的**层级面包屑（Breadcrumbs）**（如 `[员工手册 > 第五章 薪酬 > 奖金分配]`）、**文档时间**、**来源 URL** 作为 Metadata 强行拼接到该分块的文本头部 [^3][^4]。这能瞬间提升大模型的理解力。

### 3. 拒绝“一刀切”，拥抱“混合分块” (Hybrid Chunking)
现实中的文档往往是图文、表格、代码混排的。
* **策略**：先用“结构感知”提取出表格和代码块（保持原子性，绝不切断）；剩下的普通文本，如果长且复杂，走“语义分块”；如果中等长度，走“递归分割” [^4]。

### 4. 建立评估与兜底机制 (Evaluation & Guardrails)
* **度量指标**：不要凭感觉调参。建立 50 个带有标准答案的测试问题（Test Cases），每次修改 Chunk Size 或策略后，观察 **Recall@K（正确文档是否在前 K 个被召回）** [^3]。
* **护栏设计**：当检索回来的 Chunk 相似度得分都很低，或者存在时间冲突时（如同时召回了 2019 年和 2024 年的政策），系统应具备“拒绝回答”或“提示来源冲突”的兜底能力，而不是让 LLM 强行编造 [^3]。
