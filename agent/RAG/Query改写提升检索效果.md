
在 RAG（检索增强生成）系统的实际应用中，用户输入的原始问题（Query）往往是模糊的、碎片化的，或者带有口语化的简称。如果直接拿这个原始 Query 去向量数据库检索，往往会导致**检索不到相关内容**。

**Query 改写 (Query Rewriting)** 就是为了解决这个问题，它是 RAG 高级优化（Advanced RAG）中的核心步骤。

---

### 1. 核心结论 (Conclusion First)
**Query 改写**是指利用 LLM 对用户的原始输入进行重写、扩展或转换，使其更符合检索系统的索引分布，从而提高检索的**召回率 (Recall)** 和**精确度 (Precision)**。

它的本质是：**对齐用户意图与知识库语义空间。**

---

### 2. 详细拆解 (Deep Dive)

常见的改写策略有以下几种：

#### ① 查询扩展 (Query Expansion / Multi-Query)
* **原理：** 用户提问往往视角单一。利用 LLM 将一个问题生成 3-5 个等价但措辞不同的搜索词。
* **做法：** 比如用户问“Agent 怎么做？”，LLM 改写出“AI Agent 架构设计”、“如何构建大模型智能体”、“Agent 实现步骤”。
* **收益：** 克服向量检索对关键词缺失的敏感性，增加召回概率。

#### ② 假设性文档嵌入 (HyDE - Hypothetical Document Embeddings)
* **原理：** **Query（短而问）与 Document（长而答）在向量空间中天然不对称。** HyDE 让 LLM 先针对 Query 写一个“虚假的答案”，然后用这个“答案”去搜“真答案”。
* **类比：** 拿着“样板房照片”去搜“真实的毛坯房”，比拿着“设计草图”搜效果更好。

#### ③ 查询压缩/改写 (Query Compression / Rewrite for Conversation)
* **原理：** 在多轮对话中，当前 Query 往往包含代词（如“它怎么用？”）。改写器会将历史上下文注入，还原成独立完整的问题。
* **例子：**
 * 轮次1：什么是 RAG？
 * 轮次2：它有什么优点？ -> 改写为：“RAG（检索增强生成）相比微调有哪些优点？”

#### ④ 关键词提取与反向工程 (Step-back Prompting)
* **原理：** 如果问题太具体（比如具体的报错信息），LLM 先退一步提问一个更基础的概念（Step-back），从高维概念中找寻答案。

---

### 3. 面试官视角 (Interviewer's Lens)

#### 常见追问 (Follow-ups)：
1. **多路改写后，检索回来的文档太多了怎么办？**
 * *回答重点：* 必须配合 **Rerank (重排序)**。将多个 Query 搜到的结果汇聚去重，通过 Rerank 模型计算与原始意图的相关性得分。
2. **改写会增加延迟，如何权衡性能？**
 * *回答重点：* 使用更小、更快的模型（如 GPT-4o-mini 或本地端侧模型）做改写；对于简单 Query，通过意图识别跳过改写步骤。
3. **HyDE 策略如果模型产生幻觉，生成的虚假答案误导了检索怎么办？**
 * *回答重点：* HyDE 依赖于向量检索的鲁棒性。即使内容有误，其语义分布通常仍能指向正确的知识块。但在极专业领域需谨慎使用。

#### 潜在陷阱 (Pitfalls)：
* **陷阱：盲目使用所有改写技巧。**
 * *避坑：* 所有的改写都是以 **Latency（延迟）** 换取 **Accuracy（准确率）**。面试时要强调“根据业务场景选择”，例如客服场景追求快，可能只做简单的多轮纠偏；而科研助手工具有时间容忍度，可以做 HyDE。

### 4. 代码/Prompt 示例 (Python/Prompt)

以下是一个典型的 **Multi-Query Expansion** 的 Prompt 设计：

```python
QUERY_REWRITE_PROMPT = """
你是一个搜索专家。你的任务是基于用户的原始问题，生成 3 个不同维度的搜索查询词，
以便从向量数据库中检索到更全面的信息。

原始问题: {original_query}

请以 JSON 格式输出：
{{
  "expanded_queries": ["查询词1", "查询词2", "查询词3"]
}}
"""

# 示例：用户问 "DeepSeek 怎么用？"
# 结果可能产生：
# 1. "DeepSeek 模型 API 调用文档"
# 2. "DeepSeek 部署教程及参数设置"
# 3. "DeepSeek 与 GPT-4 性能对比及使用案例"
```

### 5. 快速总结 (Summary)
1. **为什么：** 解决用户表达与知识库内容之间的“语义鸿沟”。
2. **怎么做：** 扩展（多路搜索）、生成（HyDE）、纠偏（多轮对话改写）。
3. **注意：** 改写后必须接 **Rerank**，且需关注模型调用带来的**延迟增加**。
