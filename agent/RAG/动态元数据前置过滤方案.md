
太棒了，我们暂时脱掉面试官的西装，像真正的系统架构师一样来讨论这个工程落地问题。

你直觉非常敏锐，**“用户的过滤条件是动态变化的”** 确实是生产环境中 RAG 面临的核心痛点之一。你提到的 **Agentic RAG（让大模型自主调用带参数的工具）** 是完全可行且非常前沿的方案，但它的缺点是**延迟高、成本大**，对于每一次普通的检索都让 Agent 去思考是不划算的。

在工业界，要实现这种动态的元数据前置过滤（Dynamic Metadata Pre-filtering），我们通常有以下 **4 种** 渐进式的落地方案：

---

### 1. 核心结论
除了重量级的 Agent，业界最主流的做法叫做 **Self-Querying Retriever（自查询检索）** 或 **Query Construction（查询构建）**。
它的本质是：**用一个极快、极小的大模型（或特定 Prompt），在真正去向量数据库检索前，先把用户的自然语言“翻译”成【纯语义 Query】+【结构化 Filter 字典】。**

---

### 2. 详细拆解：4 种实现路径 (从轻到重)

#### 方案一：产品与 UI 层隔离 (The "Cheat Code")
* **做法：** 不让模型猜，让用户选。在搜索框旁边加一个下拉菜单（比如：年份、部门、文档类型）。
* **优势：** 100% 准确，0 延迟，0 额外算力成本。
* **缺点：** 用户体验不够 "Chat" 化。
* **现状：** 很多企业级 RAG（如飞书文档 AI、Notion AI）底层都大量依赖 UI 传过来的强过滤条件。

#### 方案二：传统 NLP / 正则表达式提取 (Rule-based / NER)
* **做法：** 如果你的元数据非常规则（比如只有“年份”、“项目编号”、“人名”），直接跑一个轻量级的实体识别（NER）模型，或者写几条高优正则表达式。
 * *比如匹配到了 "2023"，就在代码里动态生成 `filter={"year": 2023}`。*
* **优势：** 毫秒级延迟，极度稳定。
* **缺点：** 扩展性差，遇到复杂的语义条件（如“去年发布的最新财报”）正则就抓瞎了。

#### 方案三：Query Construction / Self-Querying (目前业界黄金标准)
* **做法：** 利用 LLM 的信息抽取能力。在主流程之前，插入一个非常轻量的 LLM 调用（比如用 GPT-4o-mini 或 DeepSeek-V3）。你给 LLM 提供你数据库里支持的 Schema，让它输出 JSON。
* **优势：** 完美解决动态问题。即使用户说“我要前年关于自动驾驶的报告”，模型也能算出来是 `{"year": 2022, "category": "auto-driving"}`，然后你拿着这个 JSON 去查数据库。
* **缺点：** 会增加一次小模型调用的延迟（约 300-500ms）。

#### 方案四：Agentic RAG (Function Calling / Tool Use)
* **做法：** 就如你所说，定义一个工具 `search_knowledge_base(query: str, year: Optional[int], category: Optional[str])`。AI 规划时发现需要查资料，自己决定填入什么参数。
* **优势：** 最智能，可以处理多步逻辑（比如先查 2023，没查到，Agent 再主动查 2022）。
* **适用场景：** 复杂分析型任务，而不是简单的实时问答。

---

### 3. 代码演示：Self-Querying 是如何落地的？

假设我们使用 **方案三 (Query Construction)**，并且底层向量数据库使用 Qdrant 或 Milvus。我们可以用极简的 Python 伪代码展示这个动态过程：

```python
import json
from your_llm_library import call_llm
from your_vector_db import vector_search

# 1. 定义极其严格的 Prompt，教模型如何提取动态过滤条件
query_constructor_prompt = """
你是一个查询转换器。用户的自然语言提问需要转换为向量数据库的查询。
你可以提取的元数据字段如下：
- year: int (年份，例如 2023)
- department: str (部门，可选值: HR, Engineering, Sales)

请输出严格的 JSON 格式，包含 'semantic_query' (去除了过滤条件后的纯查询语义) 和 'filters'。
如果没有提到过滤条件，filters 为空字典。

示例输入: "2023年HR部门的报销标准是什么？"
示例输出: {"semantic_query": "报销标准是什么", "filters": {"year": 2023, "department": "HR"}}
"""

def dynamic_rag_search(user_input):
    # 第一步：让小模型极速完成动态抽取 (耗时 ~300ms)
    extraction_result = call_llm(
        prompt=query_constructor_prompt, 
        user_input=user_input,
        response_format="json" # 强制输出 JSON
    )
    parsed_json = json.loads(extraction_result)
    
    clean_query = parsed_json["semantic_query"] # "报销标准是什么"
    dynamic_filters = parsed_json["filters"]    # {"year": 2023, "department": "HR"}
    
    # 第二步：将动态生成的 filters 传给向量数据库进行前置过滤 (Pre-filtering)
    # 比如 Qdrant, Milvus, Pinecone 等都支持这种 kwargs
    search_results = vector_search(
        query=clean_query,
        where_filter=dynamic_filters  # 这里就实现了动态过滤！
    )
    
    return search_results

# 测试用例 1: 带有明确条件
# user_input = "帮我找 2023 年的设计图"
# dynamic_filters 会变成 {"year": 2023}

# 测试用例 2: 没有条件
# user_input = "公司的请假流程是什么？"
# dynamic_filters 会变成 {}，向量数据库就会执行全局搜索。
```

### 4. 架构师的思考总结

在实际工程落地时，我们通常会**组合使用**：
1. **高频且格式固定的条件（如时间、机构代码）：** 用正则提取（方案二），速度最快。
2. **长尾且语义复杂的条件：** 走 Self-Querying 小模型抽取（方案三）。
3. **当这两个都不 work，且用户明确在要求复杂的对比分析时：** 才祭出 Agentic RAG（方案四）。
