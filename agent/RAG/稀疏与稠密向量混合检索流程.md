RAG（检索增强生成）中的**稀疏与稠密向量混合检索（Hybrid Retrieval）** 是目前提升检索准确率的主流方案。它的核心思想是结合**稠密检索（Dense Retrieval）** 的语义理解能力和**稀疏检索（Sparse Retrieval）** 的关键词精确匹配能力。

以下是混合检索的标准全流程详解，分为**索引构建阶段**和**查询检索阶段**。

---

### 一、核心概念对比

在深入流程前，先明确两者的区别：

-   **稠密检索 (Dense)**: 使用 Embedding 模型（如 BGE, OpenAI, M3E）将文本转化为稠密向量。
    
    -   *优势*: 擅长语义匹配（例如“怎么付款”能匹配到“支付流程”）。
        
    -   *劣势*: 对专有名词、精确 ID、生僻词匹配效果差。
        
-   **稀疏检索 (Sparse)**: 通常指基于倒排索引的关键词匹配（如 BM25, TF-IDF），或学习型稀疏向量（如 SPLADE）。
    
    -   *优势*: 擅长精确匹配（例如"SKU-12345"或“张三”）。
        
    -   *劣势*: 无法理解同义词或语义泛化。
        

---

### 二、混合检索详细流程

#### 阶段 1：数据索引构建 (Indexing Phase)

在这个阶段，需要对知识库文档进行双重处理，建立两个独立的索引。

1.  **文档预处理**:
    
    -   加载文档（PDF, Markdown, DB 等）。
        
    -   **分块 (Chunking)**: 将长文档切分为小的文本块（Chunks）。
        
2.  **双重向量化**:
    
    -   **稠密路径**: 使用 Embedding 模型将每个 Chunk 转化为**稠密向量**（例如 768 维浮点数）。
        
    -   **稀疏路径**: 对每个 Chunk 进行分词，统计词频，构建**倒排索引**（BM25）或转化为**稀疏向量**（SPLADE）。
        
3.  **存储**:
    
    -   **向量数据库** (如 Milvus, Pinecone, Weaviate): 存储 Chunk 内容 + 稠密向量 + 元数据。
        
    -   **搜索引擎/倒排索引** (如 Elasticsearch, Opensearch): 存储 Chunk 内容 + 关键词索引。
        
    -   *注：现代向量数据库（如 Milvus 2.4+, Elasticsearch 8.0+）通常支持在同一系统内同时存储稠密和稀疏索引。*
        

#### 阶段 2：查询处理 (Query Processing)

当用户提出问题（Query）时：

1.  **Query 预处理**: 清洗用户输入。
    
2.  **双重编码**:
    
    -   **稠密编码**: 使用与索引阶段**相同**的 Embedding 模型，将 Query 转化为稠密向量。
        
    -   **稀疏编码**: 对 Query 进行分词处理，提取关键词。
        

#### 阶段 3：并行检索 (Parallel Retrieval)

系统同时发起两路检索请求：

1.  **稠密检索**: 在向量数据库中进行近似最近邻搜索 (ANN)，基于向量相似度（如余弦相似度），召回 Top-K 个文档块（例如 K=50）。
    
2.  **稀疏检索**: 在搜索引擎中进行关键词匹配（如 BM25 评分），召回 Top-K 个文档块（例如 K=50）。
    
3.  **结果**: 此时得到两个独立的候选文档列表：`List_Dense` 和 `List_Sparse`。
    

#### 阶段 4：结果融合 (Fusion / Merging)

这是混合检索最关键的一步。因为两个列表的评分标准不同（向量相似度 vs BM25 分数），不能直接相加，需要归一化和融合。

**常用融合算法：**

1.  **倒数排名融合 (Reciprocal Rank Fusion, RRF)**: *（业界最常用）*
    
    -   不依赖具体分数，只依赖排名。
        
    -   公式：$Score(d) = \sum_{r \in R} \frac{1}{k + rank_r(d)}$
        
    -   逻辑：如果一个文档在两个列表中都排名靠前，它的 RRF 分数会非常高。
        
    -   优点：无需关心原始分数的分布，鲁棒性强。
        
2.  **加权求和 (Weighted Sum)**:
    
    -   先对两个列表的分数进行归一化（如 Min-Max Normalization）。
        
    -   公式：$Final\_Score = \alpha \cdot Score_{dense} + (1-\alpha) \cdot Score_{sparse}$
        
    -   缺点：需要调节 $\alpha$参数，且对分数分布敏感。
        

**输出**: 融合后得到一个统一的、去重的 Top-N 文档列表（例如 N=10）。

#### 阶段 5：重排序 (Reranking) - *可选但推荐*

融合后的列表虽然结合了两者优势，但可能仍包含噪声。为了进一步提升精度，通常加入 Cross-Encoder 重排序。

1.  **输入**: 融合后的 Top-N 文档 + 用户 Query。
    
2.  **模型**: 使用 Rerank 模型（如 BGE-Reranker, Cohere Rerank）。
    
3.  **过程**: 模型对 (Query, Document) 对进行深层交互计算，给出更精确的相关性分数。
    
4.  **输出**: 重新排序后的 Top-M 文档（例如 M=5），作为最终上下文。
    

#### 阶段 6：生成 (Generation)

1.  **Prompt 构建**: 将最终选定的 Top-M 文档内容作为 Context，与用户 Query 一起组装进 Prompt。
    
2.  **LLM 推理**: 发送给大语言模型生成答案。
    
3.  **返回**: 将答案返回给用户。

### 四、为什么要用混合检索？（场景分析）

| 场景  | 纯稠密检索 | 纯稀疏检索 | **混合检索** |
| --- | --- | --- | --- |
| **语义问答** (如："如何重置密码？") | ✅ 效果好 | ❌ 效果差 (若无 exact 关键词) | ✅ **优** |
| **精确匹配** (如："订单号 2023001 的状态") | ❌ 容易幻觉/匹配错 | ✅ 效果好 | ✅ **优** |
| **专有名词** (如：人名、产品型号、代码错误码) | ❌ 向量难以区分细微差别 | ✅ 精确命中 | ✅ **优** |
| **多语言/同义词** | ✅ 天然支持 | ❌ 需依赖词表 | ✅ **优** |

### 五、实施建议与工具

1.  **工具栈推荐**:
    
    -   **LangChain / LlamaIndex**: 框架层支持 `EnsembleRetriever` 或 `HybridSearch`。
        
    -   **数据库**: Elasticsearch (原生支持 BM25 + Vector), Milvus (支持稀疏向量插件), Weaviate (原生混合搜索), PostgreSQL (pgvector + pgsearch)。
        
2.  **参数调优**:
    
    -   **K 值**: 检索阶段召回数量通常设大一点（如 50-100），给融合和 Rerank 留余地。
        
    -   **RRF 参数 k**: 通常设为 60，用于调节排名权重的衰减速度。
        
    -   **权重**: 如果使用加权融合，通常根据业务测试，语义强则稠密权重高，精确匹配多则稀疏权重高。
        
3.  **稀疏向量的选择**:
    
    -   传统 **BM25**：实现简单，效果好，是目前的默认选择。
        
    -   学习型 **SPLADE**：也是稀疏向量，但通过模型训练增强了关键词的语义权重，效果通常优于 BM25，但计算成本稍高。
        

### 总结

RAG 混合检索的流程核心在于\*\*“双路召回，统一融合”\*\*。它通过稠密向量解决“懂意思”的问题，通过稀疏检索解决“找得准”的问题，最后通过 RRF 或 Rerank 解决“排得对”的问题，是目前企业级 RAG 系统的标准配置。
