`ConcurrentHashMap` 是 Java 并发编程中最核心的工具之一。在面试中，它几乎是必考点。要彻底理解它，我们需要从 **CAS 原理**、**Java 7 与 8 的进化**、以及**具体的加锁策略**三个维度来拆解。

---

### 一、 什么是 CAS？（底层基石）

在讲 `ConcurrentHashMap` 之前，必须先理解 **CAS (Compare And Swap，比较并交换)**。

-   **核心思想**：它是一种“乐观锁”策略。它认为冲突很少发生，所以不直接加锁，而是尝试去更新。
    
-   **操作过程**：包含三个操作数：**内存地址 V**、**预期原值 A**、**新值 B**。
    
-   如果内存地址 V 的值等于预期值 A，就把 V 的值更新为 B。
    
-   如果不相等，说明别人改过了，更新失败。此时通常会进行“自旋”（死循环重试）。
    
-   **优点**：不需要切换 CPU 状态（用户态到内核态），效率极高。
    
-   **Java 实现**：通过 `sun.misc.Unsafe` 类调用操作系统的原子指令实现。
    

---

### 二、 Java 7 vs Java 8 的底层实现

#### 1\. Java 7：分段锁（Segment）

-   **结构**：`Segment` 数组 + `HashEntry` 数组。`Segment` 继承了 `ReentrantLock`。
    
-   **原理**：把大的 HashMap 分成很多个小段（Segment），每一段配一把锁。
    
-   **缺点**：锁的粒度还是太大。如果 Segment 数量是 16，那最多只能支持 16 个线程并发。
    

#### 2\. Java 8：Node 数组 + CAS + synchronized（主流）

-   **结构**：**数组 + 链表 + 红黑树**（结构上和 HashMap 一致）。
    
-   **原理**：抛弃了 Segment，直接在数组的**每个桶（Bucket）的头节点**上加锁。
    
-   **进化**：锁的粒度从“段”精细到了“桶”，并发能力大大增强。
    

---

### 三、 Java 8 ConcurrentHashMap 如何加锁？

当我们调用 `put(key, value)` 时，线程的处理逻辑如下：

1.  **计算 Hash**：得到该 Key 应该在数组的哪个位置。
    
2.  **空桶插入（CAS）**：
    

-   如果该位置是空的，直接通过 **CAS** 尝试插入。如果成功，直接返回。
    

3.  **正在扩容（Help Resize）**：
    

-   如果发现该节点的 Hash 值是 `-1`（ForwardingNode），说明整个 Map 正在扩容，当前线程会**加入进去帮忙搬运数据**，提高效率。
    

4.  **桶内冲突（synchronized）**：
    

-   如果该位置已经有数据了（发生哈希冲突），则使用 `synchronized` **锁定该桶的头节点**。
    
-   锁住后，遍历链表或红黑树，执行插入或覆盖操作。
    

5.  **树化**：如果链表太长，依然会转为红黑树。
    

**为什么 Java 8 又用回了 synchronized？** 早期的 `synchronized` 性能差，但 Java 6 之后对其进行了大量优化（偏向锁、轻量级锁、锁自旋）。在锁粒度极细的情况下，`synchronized` 的性能并不比 `ReentrantLock` 差，且能节省大量空间。

---

### 四、 get 方法为什么不需要加锁？

这是 `ConcurrentHashMap` 性能极高的原因之一。`get` 操作是**完全无锁**的。

-   **原因**：底层 Node 节点的 `val` 和 `next` 指针都使用了 `volatile` 关键字修饰。
    
-   **作用**：`volatile` 保证了**可见性**。一个线程修改了值，另一个线程能立刻看到，不需要通过加锁来同步内存。

### 五、 面试常考点（避坑指南）

#### 1\. 为什么 ConcurrentHashMap 的 Key/Value 不能为 null？

-   **答**：为了避免**二义性**。在多线程环境下，如果 `get(key)` 返回 `null`，你无法分辨是“这个 Key 没存过”还是“这个 Key 存的值就是 null”。在单线程的 HashMap 中可以用 `containsKey()` 判断，但在多线程下，判断的一瞬间值可能就被改了。
    

#### 2\. 扩容期间还能进行读写吗？

-   **答**：可以。
    
-   **读操作**：如果读到的是已迁移完的桶，会去新数组读；如果还没迁移，就在旧数组读。
    
-   **写操作**：如果写到正在迁移的桶，线程会阻塞或加入扩容大军（协助搬运）。
    

#### 3\. size() 是如何统计的？

-   **答**：它没有用一个简单的 `int` 变量（因为高并发下写这个变量会变成瓶颈）。它借鉴了 `LongAdder` 的思想，内部维护了一个 `baseCount` 和一个 `CounterCell` 数组。多线程增加计数时，会分散到不同的 CounterCell 中，最后求和。
    

#### 4\. Java 8 相比 Java 7 最大的改进是什么？

-   **答**：
    

1.  **锁粒度更细**：从 Segment 改为 Node 桶锁。
    
2.  **数据结构优化**：引入红黑树，解决哈希冲突严重时的查询性能问题。
    
3.  **并发扩容**：支持多线程同时参与扩容。
    

---

### 总结

`ConcurrentHashMap` 就像是一个高效的自动化工厂：

-   **CAS** 是它的“快速通道”，没冲突时直接过；
    
-   **synchronized** 是它的“单人隔间”，有冲突时按桶排队；
    
-   **volatile** 是它的“全厂广播”，确保信息实时同步；
    
-   **多线程扩容** 是它的“全员协作”机制。
