`HashMap` 的底层实现是面试的“必考题”。在 Java 8 及之后的版本中，它的底层结构可以概括为：**数组 + 链表 + 红黑树**。

以下是详细的分解：

---

### 1\. 核心数据结构

`HashMap` 的本质是一个哈希表（散列表）。

-   **数组（Node<K,V>[] table）**：这是 `HashMap` 的主体，被称为“桶（Bucket）”。数组的初始长度默认是 **16**。
    
-   **链表**：当两个不同的 Key 计算出的数组下标相同时（哈希冲突），它们会以链表的形式存储在该位置。
    
-   **红黑树（Java 8 新增）**：当链表太长时，查找效率会从 $O(1)$退化为 $O(n)$。为了优化，Java 8 引入了红黑树，将查找效率提升至 $O(\log n)$。
    

---

### 2\. 存储流程（`put` 方法的过程）

当你执行 `map.put(key, value)` 时，发生了以下步骤：

1.  **计算哈希值**： 调用 Key 的 `hashCode()` 方法，并进行“扰动处理”（将高 16 位与低 16 位进行异或），目的是让 Hash 值更均匀，减少碰撞。
    
2.  **计算下标**： 使用公式 `(n - 1) & hash` 计算该 Key 应该放在数组的哪个位置（`n` 是数组长度）。 *注：这就是为什么数组长度必须是 2 的幂次方，因为这样取模运算最快。*
    
3.  **处理冲突**：
    

-   如果该位置为空，直接插入一个新 `Node`。
    
-   如果该位置不为空（发生冲突）：
    
-   **Key 相同**：直接覆盖旧的 Value。
    
-   **Key 不同**：
    
-   如果该节点已经是**红黑树**，则在树中插入。
    
-   如果该节点是**链表**，则在链表尾部插入。
    

4.  **树化（Treeify）**： 如果链表长度**超过 8**，且数组总长度**超过 64**，链表就会转成红黑树。
    
5.  **扩容检查**： 如果当前元素个数超过了 `阈值`（数组长度 \* 加载因子 0.75），就会触发 `resize` 扩容。
    

---

### 3\. 扩容机制（Resize）

-   **触发条件**：当前元素个数 > `capacity * 0.75`（默认 16 \* 0.75 = 12）。
    
-   **扩容动作**：
    

1.  创建一个容量为原来 **2 倍** 的新数组。
    
2.  **数据迁移**：遍历旧数组，将所有原始数据重新计算下标并搬到新数组。
    

-   **Java 8 的优化**：迁移时不需要重新计算哈希。由于数组长度是 2 的幂，元素要么留在“原位置”，要么移动到“原位置 + 旧数组长度”的位置。
    

---

### 4\. 几个关键参数（面试常问）

1.  **DEFAULT\_INITIAL\_CAPACITY (16)**：默认初始容量。
    
2.  **DEFAULT\_LOAD\_FACTOR (0.75)**：默认加载因子。
    

-   *为什么是 0.75？* 这是一个折中方案。如果太大（如 1.0），空间利用率高但冲突多，影响查询；如果太小（如 0.5），冲突少但空间浪费严重，且频繁扩容。
    

3.  **TREEIFY\_THRESHOLD (8)**：链表转红黑树的阈值。
    

-   *为什么是 8？* 根据泊松分布，在负载因子为 0.75 时，同一个桶中元素达到 8 个的概率极低（千万分之一）。如果真的到了 8 个，说明哈希冲突严重，需要红黑树救场。
    

4.  **UNTREEIFY\_THRESHOLD (6)**：红黑树退化为链表的阈值。扩容或删除元素时，如果树节点少于 6 个，就转回链表以节省空间。
    

---

### 5\. 为什么 HashMap 不是线程安全的？

1.  **多线程扩容死循环**：在 Java 7 中，由于采用头插法迁移链表，多线程下可能形成环形链表，导致死循环（Java 8 改用尾插法解决了死循环，但依然不安全）。
    
2.  **数据丢失**：两个线程同时 `put`，计算出相同下标，可能会导致其中一个线程的数据被另一个覆盖。
    
3.  **Size 不准**：多个线程同时增加计数，导致最终 size 与实际元素个数不符。
    

**解决方案**：

-   使用 `Collections.synchronizedMap(map)`。
    
-   使用 `ConcurrentHashMap`（推荐，性能最高）。
    

---

### 总结

`HashMap` 的设计非常精妙：

-   **数组**保证了快速定位；
    
-   **扰动函数**减少了冲突；
    
-   **链表**解决了冲突；
    
-   **红黑树**保证了极端情况下的性能；
    
-   **2 的幂次方长度**保证了取模运算的高效。
